# Production Batches — THE driving workflow

**The 4 AM tracker. Mobile-first. State machine planned → in_progress → completed | cancelled. Auto-emits inventory consumption on complete. Live yield %, COGS/bottle, gross margin rollup.**

Schema:
- `fct_production_batches` (UUID PK, tenant_id, kitchen_id FK, product_id FK, recipe_id FK to "11_somaerp".fct_recipes (the exact version used), run_date DATE, shift_start TIMESTAMP, shift_end TIMESTAMP, planned_qty NUMERIC NOT NULL, actual_qty NUMERIC, status TEXT CHECK in ('planned','in_progress','completed','cancelled') DEFAULT 'planned', currency_code CHAR(3), lead_user_id UUID, notes TEXT, properties JSONB, audit/timestamps/soft-delete) — partial unique (tenant_id, kitchen_id, product_id, run_date, shift_start) WHERE deleted_at IS NULL AND status != 'cancelled' (one active batch per kitchen+product+shift)
- `dtl_batch_step_logs` (batch_id FK, step_number INT, recipe_step_id FK nullable, name TEXT, started_at TIMESTAMP, completed_at TIMESTAMP, performed_by_user_id UUID, notes TEXT, audit/ts/soft-delete)
- `dtl_batch_ingredient_consumption` (batch_id FK, raw_material_id FK, planned_qty NUMERIC, actual_qty NUMERIC, unit_id FK, unit_cost_snapshot NUMERIC (copied from raw_material.target_unit_cost at batch start — freeze price), currency_code CHAR(3), lot_number TEXT, line_cost_actual NUMERIC GENERATED ALWAYS AS (actual_qty * unit_cost_snapshot) STORED)
- `dtl_batch_qc_results` (batch_id FK, checkpoint_id FK to dim_qc_checkpoints, outcome_id FK, measured_value NUMERIC, measured_unit_id, notes, photo_vault_key, performed_by_user_id, ts)

Views:
- `v_production_batches` — join kitchen + product + recipe + status + lead_user_email
- `v_batch_summary` — computed columns: yield_pct = actual_qty/planned_qty*100; total_cogs = SUM(line_cost_actual) + packaging (future); cogs_per_unit = total_cogs / actual_qty; gross_margin_pct = (product.default_selling_price - cogs_per_unit) / product.default_selling_price * 100; duration_min (shift_end - shift_start); step_count_completed
- `v_batch_consumption` — consumption lines joined with raw_material_name + unit_code
- `v_batch_qc_results` — QC results joined with checkpoint_name + outcome_code

Sub-feature: `70_production_batches` (5 files).

Endpoints under `/v1/somaerp/production/*`:
- GET /batches (filter kitchen_id, product_id, status, run_date range, lead_user_id)
- POST /batches (body: kitchen_id, product_id, recipe_id (if not provided, use active recipe for product), run_date default today, shift_start default now, planned_qty, lead_user_id, notes) → auto-generates step logs from recipe steps + auto-generates consumption plan from recipe ingredients × planned_qty (unit-converted)
- GET /batches/{id} (with steps + consumption + qc_results eager)
- PATCH /batches/{id} — state transitions: planned→in_progress, in_progress→completed, in_progress→cancelled, planned→cancelled. Setting status=completed requires actual_qty present; on completion auto-emit evt_inventory_movements type='consumed' for each consumption line × actual_qty (use stored ratio or the planned:actual ratio proportionally).
- DELETE /batches/{id} (soft)
- POST /batches/{id}/steps/{step_number}/start — sets started_at=now
- POST /batches/{id}/steps/{step_number}/complete — sets completed_at=now
  (NOTE: per rule "no action endpoints", could use PATCH. But since start/complete are semantic state transitions and separate from the step's own data, we'll use PATCH /batches/{id}/steps/{step_number} with body {started_at: "now"} or {completed_at: "now"} OR allow special body {action: "start"} on PATCH)
  — use PATCH /batches/{id}/steps/{step_id} with body {started_at: now_iso, completed_at: now_iso} — user can set either.
- PATCH /batches/{id}/consumption/{line_id} (body: actual_qty + lot_number)
- POST /batches/{id}/qc — record QC check result (body: checkpoint_id, outcome_id, measured_value, notes, photo_vault_key)
- GET /batches/{id}/summary — returns v_batch_summary row (yield, COGS, margin, duration)

Status transition audit: `somaerp.production.batches.started` (→in_progress), `.completed` (→completed), `.cancelled` (→cancelled).

Frontend:
- /production landing (3 cards: Batches list, New Batch, Today's Board)
- /production/batches list (filters kitchen + date + status; rows)
- /production/batches/new (form: kitchen select → product select → recipe (auto-picks active; override allowed) → planned_qty → notes. Creates batch in 'planned' state.)
- /production/batches/[id] — **MOBILE-FIRST 4 AM UI.** Info header (big status badge + kitchen + product + recipe version + lead). Two-column on desktop, stacked on mobile. Left column: Step list with big "Start Step 1"/"Complete Step 1" buttons, duration displayed live. Right column: Ingredient consumption — each line shows planned vs actual; tap to edit actual_qty + lot_number. Cost preview row updating live. Bottom: Actual Qty input + "Complete Batch" big button (requires confirm modal). Live yield %, COGS/bottle, gross margin shown at top throughout.
- /production/board (today's shift at a glance — per-kitchen card with active batches + yield rollup)
