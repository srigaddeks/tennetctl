<objective>
## Goal

Stand up the apps/somaerp backend + frontend scaffold end-to-end. By the end of this plan, somaerp boots as its own FastAPI process, exposes `GET /v1/somaerp/health` that proves the tennetctl proxy works, and the Next.js frontend renders a landing page that displays live health status. Schema namespace `"11_somaerp"` is created via somaerp's own migration. ZERO sub-feature business logic — that ships in 56-03 onwards.

## Purpose

Plans 56-03 through 56-12 (geography, catalog, recipes, QC, raw materials, procurement, production, customers, delivery, reporting) all sit on top of this scaffold. Without the backbone (config, db pool, tennetctl_client, response envelope, middleware, frontend shell), none of the vertical slices have a place to land. This plan is the "Phase 1 Plan 1" of somaerp itself — solsocial precedent (apps/solsocial/) maps 1:1, so the autonomous build is template-driven, not greenfield.

## Output

A working apps/somaerp app:
- Backend: FastAPI app with 8-file 01_core/, scripts/migrator wrapper, 00_health sub-feature (5-file pattern), feature.manifest.yaml registered as `11_somaerp`, schema `"11_somaerp"` migration
- Frontend: Next.js app with landing page, /lib/api.ts client, /types/api.ts shared types
- Tests: smoke test that asserts /v1/somaerp/health returns 200 with the response envelope and proxies tennetctl /v1/auth/me successfully (mocked tennetctl OK if live tennetctl unavailable)
- README.md at apps/somaerp/ root explaining how to run it
</objective>
