<objective>
## Goal

Ship the GEOGRAPHY VERTICAL (minus capacity, deferred to 56-06): regions read-only + locations CRUD + kitchens CRUD + service zones CRUD, end-to-end backend + frontend + real-DB tests + Soma Delights tenant seeded with `IN-TG` region, Hyderabad location, KPHB Home Kitchen, KPHB Cluster 1 service zone. By end of plan, an operator can open `/geography/kitchens` and see KPHB Home Kitchen as a real row in `"11_somaerp".fct_kitchens`, created via the API, with audit trail.

## Purpose

Geography is the foundation every other layer hangs off of: kitchens own production batches (56-10), procurement runs ingest into kitchens (56-09), customers live at locations (56-11), delivery routes ship from kitchens (56-12), recipes have per-kitchen overrides (56-07). Without this layer in place, no downstream vertical works on real data. This plan also proves the somaerp scaffold (Plan 56-02) actually serves real CRUD with real Postgres + real audit emission to tennetctl, not just a `/health` ping.

## Output

3 sub-features (`10_locations`, `15_kitchens`, `17_service_zones`) registered in feature.manifest.yaml, 5-file pattern each. One migration creating 4 tables + 4 views in schema `"11_somaerp"`. Seed YAML for `dim_regions` with `IN-TG`. Frontend admin pages for all 3 entities under `/geography/*`. Real-DB pytest suite (3 test files, 1 conftest) covering CRUD + tenant isolation + audit emission.

Capacity is **explicitly deferred** to plan 56-06 — its FKs require `fct_product_lines` (catalog, 56-04) and `dim_units_of_measure` (raw materials, 56-05) which don't exist yet. Spec doc `01_data_model/01_geography.md` defines `fct_kitchen_capacity`; this plan ships everything except that table.
