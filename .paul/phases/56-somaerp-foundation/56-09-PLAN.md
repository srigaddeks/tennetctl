# Procurement + Inventory + MRP-lite Planner

**Ships user's "raw materials needed for any given day" workflow as a POST /procurement/plan BOM-explosion endpoint.**

Schema:
- `fct_procurement_runs` (tenant_id, kitchen_id FK, supplier_id FK, run_date DATE, performed_by_user_id UUID, total_cost NUMERIC, currency_code, notes, status active/reconciled/cancelled, properties JSONB, audit/ts/soft-delete)
- `dtl_procurement_lines` (procurement_run_id FK, raw_material_id FK, quantity NUMERIC, unit_id FK, unit_cost NUMERIC, lot_number TEXT, line_cost computed, quality_grade SMALLINT, received_at TIMESTAMP, audit/ts/soft-delete)
- `evt_inventory_movements` (tenant_id, kitchen_id FK, raw_material_id FK, movement_type TEXT CHECK in ('received','consumed','wasted','adjusted','expired'), quantity NUMERIC, unit_id FK, lot_number, batch_id_ref UUID NULL (production_batches — TODO(56-10) FK), procurement_run_id FK NULL, reason TEXT, ts TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP, performed_by_user_id UUID, metadata JSONB) — evt_* append-only, NO updated_at, NO deleted_at
- View `v_inventory_current` (kitchen × raw_material → qty = SUM(received+adjusted_positive) - SUM(consumed+wasted+adjusted_negative+expired), uses units_of_measure.to_base_factor to normalize to base unit; exposes qty + base_unit)
- View `v_procurement_runs` (joins supplier + kitchen + computed line_count + computed sum_total)
- View `v_inventory_movements` (joins everything for feed display)

Sub-features:
- `60_procurement` — runs CRUD + lines (add line creates/updates run total + emits evt_inventory_movements with movement_type='received' on line.received_at set)
- `65_inventory` — movements read-only feed + current inventory view + **POST /v1/somaerp/inventory/plan** BOM-explosion endpoint

## MRP-lite planner contract

`POST /v1/somaerp/inventory/plan` body:
```json
{
  "kitchen_id": "UUID",
  "demand": [
    {"product_id": "UUID", "planned_qty": 50, "target_date": "2026-04-25"},
    ...
  ]
}
```

Response:
```json
{
  "kitchen_id": "...",
  "horizon_start": "2026-04-25",
  "horizon_end": "2026-04-25",
  "requirements": [
    {
      "raw_material_id": "UUID",
      "raw_material_name": "Spinach",
      "required_qty": 10000,
      "required_unit": "g",
      "in_stock_qty": 2500,
      "gap_qty": 7500,
      "primary_supplier_id": "UUID",
      "primary_supplier_name": "Bowenpally",
      "estimated_cost": 150.00,
      "currency_code": "INR"
    }
  ],
  "unconvertible_units": [...],
  "errors": [...]
}
```

Algorithm (server-side):
1. For each demand line, look up the active fct_recipe for that product_id (status='active') — if none, emit error "no_active_recipe" for that product_id and skip.
2. For each recipe, sum ingredient quantities × planned_qty. Convert ingredient unit to raw_material default_unit using dim_units_of_measure.to_base_factor (ratio within same dimension). If dimensions don't match, add to unconvertible_units output.
3. Roll up across all demand lines → per-raw_material required_qty.
4. Subtract v_inventory_current.qty_in_base_unit from required_qty → gap_qty.
5. Look up primary supplier via lnk_raw_material_suppliers WHERE is_primary=true + their last_known_unit_cost → estimated_cost = gap_qty × last_known_unit_cost.

Endpoints under `/v1/somaerp/procurement/*` + `/v1/somaerp/inventory/*`:
- GET /procurement/runs (filter kitchen_id, supplier_id, date range, status)
- POST /procurement/runs (create run — body: kitchen_id, supplier_id, run_date, notes)
- GET /procurement/runs/{id}
- PATCH /procurement/runs/{id} (status transitions: active→reconciled, active→cancelled)
- DELETE /procurement/runs/{id} (soft)
- GET /procurement/runs/{id}/lines
- POST /procurement/runs/{id}/lines (body: raw_material_id, quantity, unit_id, unit_cost, lot_number, quality_grade) — creates line AND emits evt_inventory_movement 'received'
- PATCH /procurement/runs/{id}/lines/{line_id} (adjust quantity/cost/lot — emit adjustment movement)
- DELETE /procurement/runs/{id}/lines/{line_id} (soft; emits compensating 'adjusted' movement)
- GET /inventory/current (filter kitchen_id, raw_material_id; returns v_inventory_current rows)
- GET /inventory/movements (filter kitchen_id, raw_material_id, movement_type, date range)
- POST /inventory/movements (manual adjustment — body: kitchen_id, raw_material_id, movement_type, quantity, unit_id, reason)
- POST /inventory/plan (MRP-lite; body above)

Frontend:
- /procurement landing (2 cards: Procurement Runs, Planner)
- /procurement/runs list + /procurement/runs/new (form: kitchen + supplier + run_date + notes; on create redirect to detail)
- /procurement/runs/[id] — detail with line add form: raw_material + quantity + unit + unit_cost + lot_number + quality_grade (1-5) + received_at. Lines table with running total.
- /inventory landing (2 cards: Current Stock, Movements Feed)
- /inventory/current — grid: kitchen filter + raw_material category filter; columns Name, Category, Current Qty, Base Unit, Reorder Alert (if below target)

- /procurement/planner — **THE MRP PAGE.** Form: Kitchen, list of demand rows (add row: Product + Planned Qty + Target Date), "Compute Plan" button. Results table: Raw Material, Required, In Stock, Gap, Primary Supplier, Est Cost. Total estimated procurement cost at bottom.
