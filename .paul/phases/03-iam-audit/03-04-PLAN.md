<objective>
## Goal
Ship the IAM read paths: (1) seed `dim_attr_defs` with 5 core attrs (org.display_name, workspace.display_name, user.email, user.display_name, user.avatar_url), (2) create three flat-shape views `v_orgs`, `v_workspaces`, `v_users` that LEFT JOIN dtl_attrs and pivot EAV rows into top-level columns using `MAX(...) FILTER (WHERE ad.code = '...')` aggregation. Each view lives in its owning sub-feature's `00_bootstrap/` per the per-sub-feature layout.

## Purpose
Phase 4+ repositories will `SELECT * FROM "03_iam"."v_orgs" WHERE id = $1` (per `.claude/rules/python.md`: reads go through views, writes go to fct_*/dtl_* tables). Without these views, every repo would repeat the same EAV pivot boilerplate. Registering attr_defs first is required — `dtl_attrs.attr_def_id` is a FK, so attr codes must exist before any feature can write attr values. Closes Phase 3.

## Output
- 1 seed YAML in the IAM feature-level bootstrap seeds dir (dim_attr_defs rows for 5 core attrs)
- 3 new SQL migrations (one per sub-feature: orgs, workspaces, users) creating `v_orgs`, `v_workspaces`, `v_users` views
- 1 pytest file verifying: attr_defs seeded; views exist; inserting a fct row + dtl_attr row returns the attr as a top-level column in the view
</objective>

  <action>
    **Pattern — `MAX(...) FILTER (WHERE ad.code = '<attr>')` pivot, one row per entity:**

    The aggregation pattern scales as attrs grow. Each view LEFT JOINs dtl_attrs (so entities without any attrs still appear with NULLs) and pivots EAV rows into top-level columns via PostgreSQL's FILTER clause.

    **Migration 1 — v_orgs** (`20260416_002_iam-orgs-view.sql`):
    ```sql
    -- UP ====
    CREATE VIEW "03_iam"."v_orgs" AS
    SELECT
        o.id,
        o.slug,
        o.is_active,
        o.is_test,
        o.deleted_at,
        o.created_by,
        o.updated_by,
        o.created_at,
        o.updated_at,
        MAX(da.key_text) FILTER (WHERE ad.code = 'display_name') AS display_name
    FROM "03_iam"."10_fct_orgs" o
    LEFT JOIN "03_iam"."21_dtl_attrs" da
        ON da.entity_type_id = 1 AND da.entity_id = o.id
    LEFT JOIN "03_iam"."20_dtl_attr_defs" ad
        ON ad.id = da.attr_def_id
    GROUP BY
        o.id, o.slug, o.is_active, o.is_test, o.deleted_at,
        o.created_by, o.updated_by, o.created_at, o.updated_at;

    COMMENT ON VIEW "03_iam"."v_orgs" IS 'Flat read shape for orgs — joins fct_orgs with dtl_attrs, pivots display_name. Phase 4+ repositories SELECT from here.';

    -- DOWN ====
    DROP VIEW IF EXISTS "03_iam"."v_orgs";
    ```

    **Migration 2 — v_workspaces** (`20260416_003_iam-workspaces-view.sql`): same pattern, entity_type_id=2. Expose: id, org_id, slug, is_active, is_test, deleted_at, created_by, updated_by, created_at, updated_at, display_name.

    **Migration 3 — v_users** (`20260416_004_iam-users-view.sql`): same pattern + resolve account_type via JOIN to `02_dim_account_types`. Expose: id, account_type (TEXT from dim), email, display_name, avatar_url, is_active, is_test, deleted_at, created_by, updated_by, created_at, updated_at.
    - DO NOT expose account_type_id — internal FK hidden
    - GROUP BY clause must include at.code

    Avoid:
    - Materialized views — refresh cost + stale data vs dtl_attrs updates
    - SELECT *.* in views — always enumerate columns (breaks if fct_ schema adds a column later, view keeps its contract)
    - Aggregating with GROUP BY o.id only — fct_orgs columns aren't in the GROUP BY then → can use `GROUP BY 1` shorthand BUT Postgres requires all non-aggregated columns to be listed; be explicit
    - Including dtl_attrs.created_at in the view — attr timestamp isn't useful at the view level; fct.updated_at already reflects change time
  </action>
