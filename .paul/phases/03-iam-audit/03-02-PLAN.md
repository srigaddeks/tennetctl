<objective>
## Goal
Register the IAM feature as the first real consumer of NCP v1: create `backend/02_features/03_iam/feature.manifest.yaml` declaring the `iam` feature (number 03, module iam, always_on true) plus 6 empty sub-feature shells (orgs, workspaces, users, roles, groups, applications) — each with their owned `"03_iam"` tables listed but no nodes/routes/pages yet. On next backend boot, the catalog upsert discovers the manifest and writes 1 feature row + 6 sub-feature rows into the `"01_catalog"` schema.

## Purpose
The catalog is empty of real features. Phase 4+ verticals will add nodes to these sub-feature shells as they ship. Establishing the shell now gives every downstream plan a consistent home — a sub-feature's `number` and directory are locked; only its `nodes: []` list grows. Also proves the catalog upsert path works end-to-end against a non-fixture feature.

## Output
- `backend/02_features/03_iam/` directory tree with `feature.manifest.yaml` at root
- 6 empty sub-feature directories (`01_orgs/` through `06_applications/`) with empty `__init__.py`
- Live catalog state: `fct_features` has 1 row (`iam`), `fct_sub_features` has 6 rows (`iam.orgs` through `iam.applications`), `fct_nodes` unchanged (no nodes yet)
- Backend lifespan log line: `"Catalog upsert: 1 features, 6 sub-features, 0 nodes"`
</objective>
