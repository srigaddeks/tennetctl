<objective>
## Goal

Make `pool` a first-class field on `NodeContext` (instead of the current `extras["pool"]` convention) and sync the NCP v1 protocol doc with two accumulated realities: the `setup` category authz bypass (§9) and the widening that control nodes may do read-only DB ops (§11 or wherever that actually lives in the doc).

## Purpose

Two deferred items from the v0.1 foundation era (STATE.md deferred-gaps list):

1. **`pool` via `extras`** — every route builder today writes `extras={"pool": pool}`; every service that emits audit via `run_node` reads `ctx.extras["pool"]`. The extras-dict pattern was a temporary workaround when NodeContext was already frozen. It's awkward, untyped, and easy to miss in new features. Promote it to a named field.

2. **Doc drift** — the NCP v1 spec and the runner's actual behavior diverged across Phase 3/4. Spec says audit scope is strict; reality has a setup-category bypass. Spec implies control nodes are compute-only; reality (Plan 04-01) widened them to do read-only DB ops. New contributors reading the spec misunderstand the runner. Fix the canonical document.

Neither is user-facing. Both unblock 39-02 (get_many pattern) + 39-03 (v1→v2 versioning demo) by leaving the protocol foundation coherent.

## Output

- `NodeContext.pool: Any = None` — new field, backward-compatible
- Every route-level NodeContext builder populates both `pool=pool` AND `extras={"pool": pool}` so new code can use `ctx.pool` without breaking existing `ctx.extras["pool"]` readers
- NCP v1 doc (`001_node_catalog_protocol_v1.md`) updated: §9 notes the setup bypass + failure-outcome bypass on audit scope; §11 (or wherever control-kind is defined) acknowledges read-only DB ops for control nodes
- No behavior change; pure plumbing + docs
</objective>
