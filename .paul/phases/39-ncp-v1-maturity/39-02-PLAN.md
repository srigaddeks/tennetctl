<objective>
## Goal

Establish the `get_many(conn, ids: list[str])` bulk-read pattern on three canonical IAM repos (orgs, workspaces, users), then document it in the backend standards so future sub-features default to it instead of re-inventing N+1.

## Purpose

Today every service that needs to resolve a list of entity IDs into rows calls `get_by_id` in a loop → N round-trips. As features compose (memberships join orgs + users, audit queries join actors + workspaces), this becomes the dominant latency tax. A single `get_many` with `WHERE id = ANY($1)` flattens it to one round-trip.

Rolling out on all ~30 repos is a future sweep. This plan ships the pattern on the three tables everyone reads against (orgs, workspaces, users) and makes the convention authoritative in the backend standards doc so new features start right.

## Output

- `get_many(conn, ids)` on orgs/workspaces/users repos — preserves v_view filtering + soft-delete guard
- Backend standards doc gains a "Bulk reads: `get_many`" section with the template
- No callers migrated (separate opportunistic cleanup)
</objective>

        if not ids:

            'SELECT <same columns as get_by_id> '
