<objective>
## Goal

Stand up the **`product_ops` feature module** end-to-end for the ingest path: Postgres schema + manifest + ingest endpoint + ingest/touches/attribution nodes + admin live-tail page. After this plan, `POST /v1/track` accepts a batch of anonymous browser events and they appear within ~1s on the `/product` admin page. SDK-side work (browser bundle, identify/merge, server SDKs, UTM dashboard) ships in 45-02 and 45-03.

## Purpose

This is the foundation of the v0.5.0 Product Ops milestone — see [CONTEXT.md](./CONTEXT.md) and [ADR-030](../../03_docs/00_main/08_decisions/030_audit_vs_product_event_streams.md). Without the schema + ingest endpoint, no SDK can land events; without the admin tail, operators can't validate that events are flowing. Everything else in the milestone (link shortener, referrals, generalized funnel engine) depends on `evt_product_events` existing as the canonical write target.

## Output

- New schema `10_product_ops` in Postgres (5 tables + 2 views, daily-partitioned `evt_*`, hot-path attribution columns on `fct_visitors` per ADR-030)
- New backend feature module `backend/02_features/10_product_ops/` with one sub-feature (`events`) and 3 nodes (`product.events.ingest`, `product.touches.record`, `product.attribution.resolve`)
- `POST /v1/track` endpoint accepting a batch (single endpoint, ADR-026 minimum-surface)
- `GET /v1/product/events` read endpoint reusing Phase 10 query DSL pattern
- `/product` admin page with LISTEN/NOTIFY live-tail (reusing Phase 10 audit explorer component, parametrized on table)
- `feature.manifest.yaml` registered in catalog at boot
- ≥80% pytest coverage on the new module
- Sidebar nav entry under "System" → "Product Ops"
</objective>

## AC-3: Privacy + cardinality guards reject misuse

```gherkin
Given a configured ingest endpoint
When the client POSTs a batch with utm_source value of length 257
Then the offending event is rejected with response 400 + error code PRODUCT_OPS.UTM_TOO_LONG
And no partial write occurs (whole batch rejected, atomic per CLAUDE.md immutability + tx=own contract)

When a workspace has already registered 500 distinct event_name values today and a 501st distinct name arrives
Then the event is rejected with response 429 + error code PRODUCT_OPS.EVENT_NAME_CAP_EXCEEDED
And the cap value is read from vault key product_ops/limits/distinct_event_names_per_day (default 500 if absent)

When a request arrives with header DNT=1
Then the event is silently dropped (response 202 with {"accepted": 0, "dropped_dnt": <count>}) — no row written, no audit row written

When IPv4 source IP 203.0.113.42 is captured server-side
Then evt_product_events.properties contains "ip_truncated": "203.0.113.0/24" (or similar) — never the raw fourth octet — unless workspace toggle product_ops/ip_truncate=false (toggle absent for v1; raw IP support deferred to 45-x)
```
