# Plan 41-02 — Dashboard Sharing + Access Control

**Phase:** 41 — Monitoring SLO + Dashboard Sharing (v0.3.0)
**Goal:** Let dashboard owners grant read-only access to dashboards either to known platform users (internal share) or to external recipients via signed tokenized URLs (public share). Every grant, view, revoke, and expiry is recorded as an append-only event so admins can audit who saw what and when.

---

<objective>

## Goal
Add a sharing layer over the Phase 13-06 dashboards sub-feature. A dashboard owner can:
1. Grant read access to N internal users (FK to `fct_users`) with optional expires_at
2. Mint a signed share token producing `/share/dashboard/{token}` — works without login, scoped to read-only, optional expiry, optional passphrase, optional viewer email capture
3. Revoke any grant immediately
4. View an event timeline showing who viewed when

The platform's existing IAM authorization is the trust root for internal grants; share tokens use a stateless HMAC + a server-side revocation list (Vault-stored signing key) for external grants.

## Purpose
Devtools dashboards are useless if you can't show them to a customer, an exec, or an on-call who isn't yet a platform user. Email screenshots lose freshness; ad-hoc screen-shares lose history. A first-class share with audit gives the same liveness as the original view, plus a record of every viewer for compliance.

## Output
- `dim_monitoring_dashboard_share_scope` (SMALLINT PK seeded: internal_user=1, public_token=2)
- `dim_monitoring_dashboard_share_event_kind` (SMALLINT PK seeded: granted=1, viewed=2, token_minted=3, token_rotated=4, revoked=5, expired=6, passphrase_failed=7)
- `fct_monitoring_dashboard_shares` (id UUID v7, org_id, dashboard_id FK fct_monitoring_dashboards, scope_id FK, granted_by_user_id FK, granted_to_user_id FK NULL, recipient_email TEXT NULL, expires_at TIMESTAMP NULL, revoked_at TIMESTAMP NULL, revoked_by_user_id FK NULL, created_at, updated_at, deleted_at) — UNIQUE (dashboard_id, granted_to_user_id) WHERE scope_id=1 AND revoked_at IS NULL AND deleted_at IS NULL
- `dtl_monitoring_dashboard_share_token` (share_id PK FK, token_hash CHAR(64) UNIQUE, key_version SMALLINT, passphrase_hash TEXT NULL, view_count INT default 0, last_viewed_at NULL) — token plaintext NEVER stored
- `evt_monitoring_dashboard_share_events` (id UUID v7, share_id FK, kind_id FK, actor_user_id FK NULL, viewer_email TEXT NULL, viewer_ip INET NULL, viewer_ua TEXT NULL, payload JSONB, occurred_at) — partitioned daily, append-only, 365-day retention
- View `v_monitoring_dashboard_shares` joining scope + grantee identity + token meta + view count + status (`active` / `expired` / `revoked`)
- `token.py`: pure `mint(share_id, key_version, secret) -> token_str`, `verify(token_str, secret_resolver) -> ShareClaim` (HMAC-SHA256 over `share_id|exp|key_version`, base64url-no-pad). Signing key resolved from Vault with rotation by key_version
- `access.py`: `can_view(conn, dashboard_id, user_id) -> bool` consulted by Phase 13-06 dashboard read routes — returns True if owner OR active internal share
- Middleware `share_token.py`: extracts `?st=...` or `Authorization: Share <token>`, verifies, sets `request.state.share_claim`, refuses if revoked/expired
- Routes (5 admin + 2 public-edge):
  - `GET /v1/monitoring/dashboards/{id}/shares` (list grants for a dashboard)
  - `POST /v1/monitoring/dashboards/{id}/shares` (create grant — internal_user OR public_token shape)
  - `GET /v1/monitoring/dashboards/{id}/shares/{share_id}` (detail + token meta, never plaintext)
  - `PATCH /v1/monitoring/dashboards/{id}/shares/{share_id}` (extend expires_at, set/clear passphrase, rotate token, revoke via {revoked_at:"now"})
  - `DELETE /v1/monitoring/dashboards/{id}/shares/{share_id}` (soft-delete; equivalent to revoke + hide)
  - `GET /v1/monitoring/dashboards/{id}/shares/{share_id}/events` (timeline)
  - Public edge: `GET /api/share/dashboard/{token}` (resolves token → returns dashboard read payload, records viewed event); `POST /api/share/dashboard/{token}/unlock` (passphrase exchange)
- Nodes: `monitoring.dashboard_share.grant_access` (effect tx=caller, audit), `monitoring.dashboard_share.revoke_access` (effect tx=caller, audit). Token mint reuses `vault.signed_token.mint` if present else local helper
- Frontend: share dialog from dashboard detail, share management page, public read-only dashboard renderer at `/share/dashboard/[token]`
- ≥20 pytest + 1 Robot E2E

## AC-1: Internal user grant + read enforcement
```gherkin
Given dashboard D owned by user A in org O
And user B exists in org O without explicit dashboard access
When A POSTs /v1/monitoring/dashboards/D/shares {scope:"internal_user", granted_to_user_id:B, expires_at:null}
Then a fct_monitoring_dashboard_shares row is created with scope_id=1
And evt_monitoring_dashboard_share_events records kind=granted with actor=A
And user B's GET /v1/monitoring/dashboards/D returns 200 (was 403)
And user C (no grant) still gets 403
When A PATCHes the share with {revoked_at:"now"}
Then evt records kind=revoked
And user B's next GET /v1/monitoring/dashboards/D returns 403 within the next request (no cache lag)
And UNIQUE constraint blocks A from creating a duplicate active grant for B on the same dashboard
```

## AC-2: Public token mint + verify + view event
```gherkin
Given dashboard D and a POST /v1/monitoring/dashboards/D/shares {scope:"public_token", expires_at:"+7d", recipient_email:"exec@acme.com"}
Then the response contains {share_id, token} ONCE — token is the only place plaintext appears
And dtl_monitoring_dashboard_share_token stores token_hash = sha256(token), key_version, view_count=0
And evt records kind=token_minted with payload.recipient_email
When an unauthenticated request hits GET /api/share/dashboard/{token}
Then share_token middleware verifies HMAC against Vault key key_version
And returns the dashboard read payload (panels, queries — same shape as authenticated GET)
And evt records kind=viewed with viewer_ip + viewer_ua + viewer_email=null
And dtl.view_count increments to 1, dtl.last_viewed_at set
When the same token is used after expires_at
Then 410 Gone is returned, evt records kind=expired ONCE per token (not per request)
When the share is PATCHed {revoked_at:"now"} before expiry
Then subsequent token use returns 403, evt records kind=revoked at PATCH time
```

## AC-3: Passphrase-protected public share
```gherkin
Given a public_token share created with {passphrase:"hunter2"}
Then dtl_monitoring_dashboard_share_token.passphrase_hash is bcrypt(passphrase)
And token plaintext alone returns 401 with body indicating passphrase required
When POST /api/share/dashboard/{token}/unlock {passphrase:"wrong"} called
Then 401 returned, evt records kind=passphrase_failed with viewer_ip
And after 5 failures within 10 minutes from same IP, the share auto-revokes (revoked_at set, evt kind=revoked, payload.reason="brute_force")
When POST .../unlock {passphrase:"hunter2"}
Then a short-lived session cookie (15min) is set; subsequent GETs succeed without re-prompt
```

## AC-4: Sharing event timeline + audit export
```gherkin
Given a share with mixed events (granted, viewed×3, token_rotated, revoked)
When GET /v1/monitoring/dashboards/{id}/shares/{share_id}/events?from=...&to=...
Then events are returned in chronological order with kind, actor_user_id (or viewer_email), viewer_ip, occurred_at
And the timeline is append-only — no DELETE or PATCH endpoint exists for events
And the audit feature (Phase 04) sees a higher-level emission per state-changing kind (granted, revoked, token_rotated, expired) tagged action="dashboard_share.{kind}"
And the platform admin can filter the events page by viewer_email or viewer_ip across ALL shares of one dashboard
```

## DO NOT CHANGE
- Phase 13-06 dashboard CRUD/storage shape (only addition: `access.can_view` consulted in read routes)
- IAM auth middleware (share token middleware mounts only on `/api/share/dashboard/*` and is additive)

## SCOPE LIMITS
- Read-only sharing only — no shared editing or commenting in v1
- No team/group grants (one user per internal grant; multi-grant by repeating the call)
- No "share via email" delivery — recipient_email is captured for audit, not sent (Notify integration is future scope)
- No download/export capture in events (PDF/CSV download not in scope here)
- No watermarking of shared dashboards
- No SSO/SAML federation for public viewers — passphrase is the only secondary factor
- No incident/SLO sharing in this plan — same pattern can be ported later
- Plan 41-01 (SLO) is fully orthogonal: dashboard sharing works whether SLOs exist or not
