# Plan 41-01 — SLO Definition & Tracking

**Phase:** 41 — Monitoring SLO + Dashboard Sharing (v0.3.0)
**Goal:** Add a Service Level Objective layer over the existing monitoring metric/alert pipeline. Operators define SLOs (target percentage, time window, indicator query), and the platform continuously computes attainment, error budget remaining, and multi-window burn rate — emitting an alert when burn rate predicts exhaustion before window end.

---

<objective>

## Goal
Introduce a first-class SLO sub-feature in monitoring. An SLO is a contract: "99.9% of indicator events succeed over rolling 30 days." The system stores the definition, runs an evaluator worker that walks `evt_monitoring_metric_samples` (Phase 13-04), computes good/total counts per window, derives `attainment_pct`, `error_budget_remaining_pct`, and a Google SRE multi-window burn rate (1h, 6h, 24h, 3d). When a fast-burn or slow-burn threshold trips, a synthetic alert event is created via the Phase 13-08 alert pipeline so existing escalation, actions, and incident grouping (Plan 40-03) all reuse without forks.

## Purpose
Threshold alerts answer "is it broken right now?" — SLOs answer "are we shipping reliably enough to keep our promise?" Without SLOs, teams either over-page (every blip) or under-page (only outages, miss slow degradation). Burn rate gives a leading indicator: "at the current error rate you'll exhaust the 30-day budget in 4 hours" — actionable hours before classic threshold alerts would notice.

## Output
- `dim_monitoring_slo_indicator_kind` (SMALLINT PK, seeded: ratio=1, threshold=2, latency_pct=3)
- `dim_monitoring_slo_window_kind` (SMALLINT PK, seeded: rolling_7d=1, rolling_28d=2, rolling_30d=3, calendar_month=4, calendar_quarter=5)
- `fct_monitoring_slos` (id UUID v7, org_id, workspace_id, name TEXT, slug TEXT, indicator_kind_id FK, window_kind_id FK, target_pct NUMERIC(6,4) CHECK 0<x<100, severity_id FK dim_alert_severity, owner_user_id FK NULL, is_active BOOL, created_at, updated_at, deleted_at) — UNIQUE (org_id, slug) WHERE deleted_at IS NULL
- `dtl_monitoring_slo_indicator` (slo_id PK FK, good_query TEXT, total_query TEXT, threshold_metric_id FK NULL, threshold_value NUMERIC NULL, threshold_op TEXT CHECK IN ('lt','lte','gt','gte','eq'), latency_percentile NUMERIC NULL) — EAV-flavored fixed-detail row, fields populated by indicator_kind
- `dtl_monitoring_slo_burn_thresholds` (slo_id PK FK, fast_window_seconds INT default 3600, fast_burn_rate NUMERIC default 14.4, slow_window_seconds INT default 21600, slow_burn_rate NUMERIC default 6.0, page_on_fast BOOL default true, page_on_slow BOOL default true) — Google SRE defaults
- `evt_monitoring_slo_evaluations` (id UUID v7, slo_id FK, window_start, window_end, good_count BIGINT, total_count BIGINT, attainment_pct NUMERIC(7,5), budget_remaining_pct NUMERIC(7,5), burn_rate_1h NUMERIC, burn_rate_6h NUMERIC, burn_rate_24h NUMERIC, burn_rate_3d NUMERIC, evaluated_at) — partitioned daily, 90-day retention
- `evt_monitoring_slo_breaches` (id UUID v7, slo_id FK, breach_kind TEXT CHECK IN ('budget_exhausted','fast_burn','slow_burn','target_missed'), burn_rate NUMERIC NULL, alert_event_id UUID NULL FK, occurred_at, resolved_at NULL)
- View `v_monitoring_slos` joining indicator + thresholds + latest evaluation + computed `status` (`healthy` / `warning` / `breaching`)
- `budget.py`: pure `compute_budget(target_pct, good, total) -> (attainment, budget_remaining_pct, budget_remaining_events)`
- `burn_rate.py`: pure `compute_burn_rate(error_rate_observed, target_error_rate, window_seconds, full_window_seconds) -> float` returning multiplier (1.0 = exactly on budget)
- `slo_evaluator_worker.py`: ticks every 60s; for each active SLO loads samples in widest needed window; writes one `evt_monitoring_slo_evaluations` row; if burn or budget breach detected, creates a synthetic `fct_monitoring_alert_rules` virtual rule (or reuses pre-seeded `slo:{id}` rule) and emits an alert event so Plan 40-03 grouper opens an incident
- Routes (5): list, create, detail, patch, soft-delete; +1 read-only `GET /v1/monitoring/slos/{id}/budget?window=...` for the budget chart, +1 `GET /v1/monitoring/slos/{id}/evaluations?from&to` for the time-series chart
- Nodes: `monitoring.slo.evaluate` (effect tx=own, worker entry), `monitoring.slo.burn_alert` (effect tx=caller, emits synthetic alert)
- ≥18 pytest + 1 Robot E2E

## AC-1: SLO CRUD + indicator wiring
```gherkin
Given a workspace with metrics ingested for the past 30 days
When operator POSTs /v1/monitoring/slos with {name:"Checkout Success", slug:"checkout-success", indicator_kind:"ratio", window_kind:"rolling_30d", target_pct:99.9, indicator:{good_query:"checkout_success_total", total_query:"checkout_attempt_total"}}
Then a row is created in fct_monitoring_slos and dtl_monitoring_slo_indicator
And dtl_monitoring_slo_burn_thresholds is seeded with Google SRE defaults (fast=14.4 over 1h, slow=6.0 over 6h)
And GET /v1/monitoring/slos returns the SLO with status="healthy" until first evaluation
And PATCH /v1/monitoring/slos/{id} {target_pct:99.95} updates the target
And DELETE soft-deletes; subsequent GET excludes it
```

## AC-2: Error budget calculation
```gherkin
Given an SLO with target_pct=99.9 over rolling_30d, good=999500, total=1000000
When budget.compute_budget(99.9, 999500, 1000000) runs
Then attainment_pct = 99.95000
And budget_remaining_pct = 50.0  # half the 0.1% error budget consumed
And budget_remaining_events = 500  # of the 1000 allowed errors

Given good=998000, total=1000000
Then attainment = 99.8, budget_remaining_pct = -100.0 (over budget by 100%)
And status flips to "breaching"
```

## AC-3: Multi-window burn rate
```gherkin
Given an SLO with target_pct=99.9 (allowed_error_rate=0.001) over rolling_30d
And the past 1h shows error_rate=0.0144 (14.4× budget)
When burn_rate.compute_burn_rate(0.0144, 0.001, 3600, 2592000) runs
Then it returns 14.4

Given the burn_rate_1h crosses the SLO's fast_burn threshold (14.4)
When the evaluator worker writes the next evt_monitoring_slo_evaluations row
Then evt_monitoring_slo_breaches inserts a row with breach_kind="fast_burn", burn_rate=14.4
And monitoring.slo.burn_alert node fires emitting a synthetic alert_event tied to the seeded virtual rule "slo:{slo_id}"
And Plan 40-03 grouper opens an incident with title "SLO fast burn: Checkout Success"
And no duplicate breach row is created while the existing breach is unresolved
```

## AC-4: Worker tick + evaluation persistence
```gherkin
Given 3 active SLOs and slo_evaluator_worker running on 60s tick
When one tick completes
Then 3 new evt_monitoring_slo_evaluations rows exist (one per SLO)
And each row has good_count, total_count, attainment_pct, budget_remaining_pct, and all four burn rates populated
And the worker holds an advisory lock keyed on slo_id (no double-eval on parallel pods)
And v_monitoring_slos returns the latest evaluation per SLO via lateral join
And UI page /monitoring/slo renders the budget chart and burn-rate gauge using these rows
```
