**Phase:** 45 — GDPR DSAR  
**Milestone:** v1.0 Self-Service  
**Status:** PENDING-APPLY (deferred to v1.0)  
**Type:** Full-stack vertical (operator dashboard + self-service endpoints + Playwright E2E)

## Scope

Add three UI surfaces for DSAR:

1. **Operator Dashboard** (`/iam/dsar`) — manage DSAR requests, view history, trigger exports/deletes
2. **Self-Service Endpoints** (`POST /v1/account/data-export`, `POST /v1/account/delete-me`) — end users initiate their own requests
3. **Account Settings Page** (`/account/data`) — users check export status, download results

Deferred to v1.0 because it requires additional complexity (2FA verification, 30-day recovery window, email notifications) not needed for v0.8.0 compliance gate.

## Technical Approach

### Backend Routes (45-02 addition to 45-01)

#### Self-Service Endpoints

1. **`POST /v1/account/data-export`**  
   - Requires authenticated session + 2FA check (if enabled)  
   - Input: `{confirm: true}` (user acknowledges data will be shared)  
   - Returns: `{job_id, estimated_completion_minutes}`  
   - Creates job with subject_user_id = current user (from session)

2. **`POST /v1/account/delete-me`**  
   - Requires authenticated session + 2FA verification + password re-entry  
   - Input: `{password, confirmation_code}` (from 2FA challenge)  
   - Returns: `{job_id, recovery_window_days: 30}`  
   - Creates deletion job with is_permanent=false (soft-delete user, hard-erase in 30d)

3. **`GET /v1/account/export-status`**  
   - Poll export job for current user  
   - Returns: `{status, completed_at, download_url}` (signed URL with 7-day TTL)

4. **`POST /v1/account/cancel-deletion`**  
   - Cancel deletion job within 30-day window  
   - Sets deletion job status=cancelled, restores user.deleted_at = NULL  
   - One-time use (cannot re-cancel after cancellation)

#### Deletion Job Behavior

- Day 0: user requests delete, user.deleted_at = NOW(), emails sent, job status=requested  
- Day 0-29: user can call `POST /v1/account/cancel-deletion` to restore account  
- Day 30: background job `_purge_scheduled_deletes` runs, hard-erases all data, job status=completed  
- After day 30: restoration impossible

#### Operator Routes (45-02 addition to 45-01)

1. **`POST /v1/iam/dsar/{user_id}/export`**  
   - Operator-initiated export  
   - Returns: `{job_id}`

2. **`POST /v1/iam/dsar/{user_id}/delete`**  
   - Operator-initiated delete (immediate, no recovery window)  
   - Returns: `{job_id}`

3. **`GET /v1/iam/dsar/jobs`**  
   - List all DSAR jobs in workspace  
   - Filters: user_id, job_type, status, date range  
   - Returns: paginated list with actor/subject names

### Backend Schema Extensions (45-02 addition to 45-01)

Add to evt_dsar_jobs:

- `is_user_initiated BOOLEAN` — true if self-service, false if operator-triggered  
- `recovery_window_end_at TIMESTAMP` — null for exports, day 30 for deletions  
- `is_permanent BOOLEAN` — true if operator deletion (immediate), false if user deletion (30d window)  
- `cancellation_requested_at TIMESTAMP` — null unless cancelled by user

Background job table `evt_scheduled_deletes`:

- `job_id UUID v7 PRIMARY KEY` (FK to evt_dsar_jobs)  
- `purge_at TIMESTAMP` (current_timestamp + 30 days)  
- `purged_at TIMESTAMP` (null until completed)  
- No updated_at (append-only)  
- created_at, created_by, is_test

### Email Templates (Notify integration)

1. **Export Ready Notification**  
   - Sent when export job completes  
   - Body: "Your data export is ready. Download link: [URL] (expires in 7 days)"  
   - Template: `iam.dsar.export_ready`  
   - Variables: {job_id, download_url, expires_at, record_count}

2. **Deletion Initiated Notification**  
   - Sent when user requests delete  
   - Body: "Your account deletion has been scheduled for [DATE]. You can cancel anytime before then: [CANCEL_LINK]"  
   - Template: `iam.dsar.deletion_initiated`  
   - Variables: {purge_date, cancel_link, days_remaining}

3. **Deletion Final Notice**  
   - Sent on day 29, reminder of final day to cancel  
   - Template: `iam.dsar.deletion_final_notice`  
   - Variables: {cancel_deadline_hours}

## Implementation Notes

1. **Recovery Window Semantics**  
   - User deletion: soft-delete user.deleted_at immediately, background job hard-erases on day 30  
   - Operator deletion: hard-delete immediately (no recovery)  
   - Cancelled deletion: restore user.deleted_at = NULL, mark job status=cancelled (one-time only)

2. **Email Delivery Timing**  
   - Deletion initiated: instant  
   - Export ready: instant (when job completes)  
   - Deletion final notice: day 29 (scheduled job)  
   - Deletion completed: instant (after hard-erase)

3. **Performance**  
   - Export can take minutes for large datasets; always async  
   - Deletion is fast (single transaction); can be sync but deferred to background for safety  
   - 30-day recovery requires background scheduler (APScheduler or similar)

4. **Security**  
   - All self-service endpoints check `ctx.session.user_id` matches target user  
   - Operator endpoints check `ctx.session.user.role` includes admin  
   - Deletion confirmation requires re-entry of password + 2FA  
   - Download URLs signed (vault integration) + short-lived (7 days)
