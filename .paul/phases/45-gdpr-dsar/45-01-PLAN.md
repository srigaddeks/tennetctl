# Plan 45-01 — GDPR DSAR Backend (Export + Delete)

**Phase:** 45 — GDPR DSAR  
**Milestone:** v0.8.0 OSS Compliance  
**Status:** PENDING-APPLY  
**Type:** Backend vertical (schema + service + routes + audit)  
**Depends on:** Phase 03 (IAM), Phase 04 (Audit), Phase 06 (Notify)

---

## Scope

Enable users to export and delete their account data — a legal requirement for GDPR/CCPA compliance. Scope is limited to **user account data** (org/workspace/user records + audit events + session history + notification subscriptions), NOT cross-feature data like feature flags or monitoring metrics.

This is operator-triggered in v0.8.0 (no self-service UI yet); self-service portal is deferred to v1.0.

---

## Success Criteria

- [ ] `POST /v1/dsar/export-request` — creates async job, returns job_id  
- [ ] `GET /v1/dsar/jobs/{id}` — poll status + download signed URL when complete  
- [ ] Export includes: user record + all orgs (with acls) + all workspaces + all sessions + audit events (where actor_id = user) + notification subscriptions  
- [ ] Export format: JSON Lines (one record per line, streaming-safe)  
- [ ] `POST /v1/dsar/delete-request` — hard-delete all user data atomically; soft-delete user record  
- [ ] Delete is idempotent (re-running does nothing)  
- [ ] Both ops write audit events: `iam.dsar.exported` / `iam.dsar.deleted` with actor (operator) + subject (target user) + counts  
- [ ] Rate limiting: 10 DSAR ops per workspace per hour (via existing PG-native rate_limit from Phase 38)  
- [ ] All data removed from Postgres (hard-delete evt_audit + dtl_* + lnk_* rows for target user)  
- [ ] pyright exit 0 + pytest green (80%+ coverage)

## Technical Approach

### Schema (Migration 071)

1. **`evt_dsar_jobs`** — new immutable event table  
   - `job_id UUID v7 PRIMARY KEY`  
   - `job_type TEXT` (export | delete)  
   - `actor_user_id UUID v7 NOT NULL` (who triggered it)  
   - `subject_user_id UUID v7 NOT NULL` (whose data is being accessed/deleted)  
   - `org_id UUID v7 NOT NULL` (workspace scope)  
   - `status TEXT` (requested | in_progress | completed | failed)  
   - `row_counts JSONB` (e.g. `{"org": 3, "workspace": 5, "audit_events": 127}`)  
   - `error_message TEXT` (null unless status=failed)  
   - `result_location TEXT` (vault key like `iam.dsar.export.{job_id}` where JSON is stored)  
   - `created_at TIMESTAMP`, `completed_at TIMESTAMP` (null until done)  
   - No `updated_at` (immutable append-only)  
   - `created_by NOT NULL` (operator, for audit scope)  
   - `is_test BOOLEAN` (false)

2. **`v_dsar_jobs`** view — org_id scope  
   - Exposes actor_user_id → display_name, status, counts, timestamps

### Sub-feature: `iam.dsar`

Five files:

1. **`schemas.py`**  
   - `DsarExportRequest(BaseModel)` — job_id, actor_user_id, subject_user_id, org_id, status  
   - `DsarJobResponse(BaseModel)` — job_id, status, completed_at, result_location, row_counts, error  
   - `DsarDeleteRequest(BaseModel)` — same, status={requested|in_progress|completed|failed}

2. **`repository.py`**  
   - `create_dsar_job(conn, actor_user_id, subject_user_id, org_id, job_type) -> dict` — INSERT evt_dsar_jobs  
   - `get_dsar_job(pool, job_id, org_id) -> dict | None` — SELECT with org scope  
   - `update_dsar_job_status(conn, job_id, status, row_counts=None, error=None)` — atomically update  
   - `export_user_data(pool, subject_user_id, org_id) -> dict` — aggregate JSON from all tables  
   - `delete_user_data(conn, subject_user_id, org_id) -> dict` — hard-delete cascade, return counts

3. **`service.py`**  
   - `create_export_request(pool, ctx, subject_user_id, org_id) -> job_id` — check rate limit, create job, dispatch async task  
   - `create_delete_request(pool, ctx, subject_user_id, org_id) -> job_id` — same for delete  
   - `poll_dsar_job(pool, ctx, job_id) -> dict` — retrieve job status + download URL  
   - Async handler (via NATS or background thread): `_process_export_job(pool, job_id)` — calls repo.export_user_data, stores JSON to vault, updates evt_dsar_jobs.result_location  
   - Async handler: `_process_delete_job(pool, job_id)` — calls repo.delete_user_data, emits audit, updates status

4. **`routes.py`** (APIRouter)  
   - `POST /v1/dsar/export-request` → `create_export_request()`  
   - `POST /v1/dsar/delete-request` → `create_delete_request()`  
   - `GET /v1/dsar/jobs/{id}` → `poll_dsar_job()` (includes signed URL to vault if export complete)  
   - `GET /v1/dsar/jobs?org_id=X` → list (filtered by org + workspace scope)

5. **`__init__.py`** — empty

### Export Data Model

One JSON object per user, streaming-friendly format:

```json
{
  "user": {...},
  "organizations": [{org_id, name, display_name, created_at}],
  "workspaces": [{workspace_id, org_id, name, created_at}],
  "workspace_members": [{workspace_id, user_id, role, created_at}],
  "sessions": [{session_id, created_at, expires_at, last_activity_at, ip_address, user_agent}],
  "audit_events": [{event_key, outcome, category, created_at, ...}],
  "notification_subscriptions": [{sub_id, channel, created_at}]
}
```

Stored in vault at `iam.dsar.{job_id}` with 90-day expiration (secure download link TTL).

### Delete Cascade

Hard-delete from all IAM tables for target_user_id:

1. Soft-delete `fct_users` (set deleted_at)  
2. Hard-delete from:  
   - `dtl_user_attrs` (WHERE user_id = X)  
   - `lnk_workspace_members` (WHERE user_id = X)  
   - `lnk_org_members` (WHERE user_id = X) [if exists]  
   - `evt_audit` rows where actor_id = X (audit-of-user-activity)  
   - `fct_sessions` (WHERE user_id = X) [soft-delete: set logged_out_at if not already]  
   - `dtl_notification_subscriptions` (WHERE user_id = X)

All in a single transaction. Row counts captured and stored in evt_dsar_jobs.row_counts.

### Audit Emission

Two new node kinds for the catalog:

- `iam.dsar.exported` (effect node, kind=effect, emits_audit=true)  
  - Input: {job_id, subject_user_id, org_id, row_counts}  
  - Emits to audit: `iam.dsar.exported` with counts in attributes  
- `iam.dsar.deleted` (effect node, kind=effect, emits_audit=true)  
  - Input: {job_id, subject_user_id, org_id, row_counts}  
  - Emits to audit: `iam.dsar.deleted` with counts in attributes

Rate-limit node reuse:
- `core.rate_limit.check` (request node from Phase 38)  
  - Config: key=`dsar.{org_id}`, window=3600s, limit=10

### Rate Limiting

Enforce at route handler level:

```python
@router.post("/v1/dsar/export-request")
async def export_request(org_id: str, subject_user_id: str, conn):
    # Check rate limit via node
    rl = await run_node(pool, "core.rate_limit.check", ctx, {
        "key": f"dsar.{org_id}",
        "action": "export",
        "limit": 10,
        "window_seconds": 3600
    })
    if not rl["allowed"]:
        raise HTTPException(429, "Rate limit exceeded")
    
    job_id = await service.create_export_request(...)
    return {"ok": True, "data": {"job_id": job_id}}
```

## Implementation Notes

1. **Async Job Processing**  
   - Use background task queue (NATS or threading) to avoid blocking export HTTP response  
   - Job status transitions: requested → in_progress → completed (or failed)  
   - Result stored in vault for secure download (signed URL with TTL)

2. **Vault Integration**  
   - Export JSON stored at `iam.dsar.{job_id}` with 90-day expiration  
   - Download requires org_id scope check

3. **Audit Scope**  
   - Both export + delete must carry actor_user_id (who triggered), subject_user_id (whose data), org_id  
   - No user_id bypass for DSAR ops (always audit)

4. **Performance**  
   - Export query uses chunked reads to avoid OOM on large datasets  
   - Delete uses single transaction (tight, no intermediate commits)

5. **Idempotency**  
   - Export: re-run same job_id returns cached result_location if completed  
   - Delete: checks deleted_at before cascading; idempotent key in evt_dsar_jobs row
