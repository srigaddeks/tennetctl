# Plan 40-02 — Action Templates (Webhook + Email + Slack)

**Phase:** 40 — Monitoring Alerting (v0.3.0)
**Goal:** Reusable, parameterized action templates that turn an alert payload into a rendered HTTP/email/Slack delivery. Adds a per-template signing secret so receivers can verify authenticity, plus a delivery log for retry/audit.

---

<objective>

## Goal
Introduce action templates as a first-class object: a template defines kind (webhook|email|slack), a body template (Jinja2 sandboxed), headers, and a signing secret. Escalation steps and alert rules can reference an action template by id. The action dispatch worker renders + delivers + logs each action with retry-on-failure.

## Purpose
Phase 13 + Phase 40-01 use Notify for human-facing channels. Action templates cover machine-facing destinations (webhook to runbook automation, generic Slack incoming-webhook, raw email to a list) where the Notify template engine is too opinionated. Operators need to ship custom JSON bodies, custom headers, and HMAC-signed payloads.

## Output
- `fct_monitoring_action_templates` (id UUID v7, org_id, name, kind_id FK dim_action_kind, target_url TEXT NULL, target_address TEXT NULL, body_template TEXT, headers_template JSONB DEFAULT '{}', signing_secret_vault_ref TEXT NULL, retry_policy JSONB, is_active, created_at, updated_at, deleted_at)
- `dim_monitoring_action_kind` (SMALLINT PK, seeded: webhook=1, email=2, slack=3, ms_teams=4)
- `evt_monitoring_action_deliveries` (id UUID v7, template_id FK, alert_event_id FK NULL, escalation_state_id FK NULL, attempt SMALLINT, status_code INT NULL, request_payload_hash TEXT, response_excerpt TEXT NULL, error_excerpt TEXT NULL, started_at, completed_at NULL, succeeded_at NULL) — partitioned daily, 30-day retention
- Renderer: Jinja2 sandboxed env with allow-list filters (no `import`, no attribute access on dangerous objects)
- Dispatchers per kind, each implementing async `dispatch(rendered, headers, signing_secret) -> DeliveryResult`
- Signing: HMAC-SHA256 over body, header `X-Tennet-Signature: t={ts},v1={hmac_hex}`; secret stored in Vault, referenced by `signing_secret_vault_ref`
- `action_dispatch_worker.py` consumes a Postgres LISTEN channel `monitoring_action_dispatch`; service.py issues NOTIFY when an action is enqueued
- Retry: exponential backoff per template's retry_policy {max_attempts, base_seconds, max_seconds}; failed deliveries logged but never silently dropped
- Nodes: `monitoring.actions.template_create/update/delete`, `monitoring.actions.render` (control), `monitoring.actions.dispatch` (effect tx=own)
- Escalation step kind extension: `notify_action` kind added to `dim_monitoring_escalation_step_kind` (id=6) — back-compat additive
- Alert rule `action_template_ids JSONB` array column added (additive — fires the action(s) on every transition)
- ≥18 pytest + 1 Robot E2E

</objective>

## AC-1: Action template CRUD
```gherkin
Given POST /v1/monitoring/action-templates with {name, kind, target_url|target_address, body_template, headers_template?, signing_secret_vault_ref?, retry_policy?}
Then a row is created in fct_monitoring_action_templates
And body_template is parsed at create-time; syntax errors return 400 RENDER_PARSE_ERROR with line/column
And kind=webhook|slack|ms_teams requires target_url; kind=email requires target_address
And signing_secret_vault_ref must resolve to an existing vault secret or 400 VAULT_REF_NOT_FOUND
And nodes monitoring.actions.template_create/update/delete exist with audit
```

## AC-2: Renderer determinism + sandboxing
```gherkin
Given body_template="alert {{rule_name}} value={{value}} labels={{labels|tojson}}"
And input vars={"rule_name":"high_cpu","value":0.92,"labels":{"host":"x"}}
Then renderer.render(template, vars) returns identical output across calls
And these patterns are rejected at parse-time:
  - {% import ... %}
  - {{ obj.__class__ }}
  - {{ ''.__class__.__mro__ }}
And only allow-listed filters are exposed: tojson, length, upper, lower, default, replace, round, int, float
And render call is bounded: max 64KB output, max 50ms wall-clock — exceed → RENDER_LIMIT error logged in delivery row
```

## AC-3: Webhook dispatcher + signing
```gherkin
Given action template kind=webhook, target_url=https://example.com/hook, signing_secret_vault_ref="vault://secret/team/hook"
When dispatch is invoked with rendered body B
Then HTTP POST is sent to target_url with:
  - Content-Type: application/json (override-able by headers_template)
  - X-Tennet-Signature: t=<unix_ts>,v1=<hmac_sha256_hex(secret, "<unix_ts>." + B)>
  - X-Tennet-Delivery-Id: <delivery_id>
  - User-Agent: tennetctl-monitoring/1.0
And TLS verification is on by default (cannot be disabled per template)
And response body is truncated to 4KB and stored in evt_monitoring_action_deliveries.response_excerpt
And 2xx → succeeded_at set; 4xx-non-429 → permanent failure (no retry); 5xx, 429, network errors → retry per policy
```

## AC-4: Slack dispatcher
```gherkin
Given kind=slack with target_url=https://hooks.slack.com/services/T/B/X
And body_template renders a valid Slack blocks JSON ({blocks:[...]} or {text:"..."})
Then dispatch posts the rendered JSON to the incoming webhook
And HTTP 200 with response body "ok" → succeeded_at; any other response → retry per policy
And no signing header is added (Slack webhooks do not validate HMAC)
And helper var {{slack_color}} is injected from severity (info=#36a64f, warn=#ffae42, error=#e01e5a, critical=#7a0000)
```

## AC-5: Email dispatcher
```gherkin
Given kind=email with target_address=ops@example.com
Then renderer produces {subject, body_text, body_html?} via a multipart body_template using {% block subject %} / {% block text %} / {% block html %}
And dispatch routes through the existing Notify SMTP transport (Phase 11 worker) — no direct SMTP code in monitoring
And delivery row references the notify delivery_id for cross-feature traceability
```

## AC-6: Retry + delivery log
```gherkin
Given a webhook target returns 502 on first attempt
Then evt_monitoring_action_deliveries records attempt=1 with error_excerpt
And worker re-enqueues with delay = min(base_seconds * 2^(attempt-1), max_seconds) jitter ±20%
And on attempt=max_attempts with failure: succeeded_at remains NULL, completed_at set, no further attempts
And metric monitoring.actions.deliveries_total{kind, outcome} + duration histogram emitted
And GET /v1/monitoring/action-deliveries?template_id=X&status=failed returns the failed deliveries
```

## AC-7: Integration with rules + escalation

```gherkin
Given an alert rule with action_template_ids=["t1","t2"]

And escalation step kind=notify_action with target_ref={action_template_id} dispatches the action at that step (advancing escalation worker calls into action service)
```
