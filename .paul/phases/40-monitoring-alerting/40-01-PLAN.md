# Plan 40-01 — Escalation Policies + On-Call Schedules

**Phase:** 40 — Monitoring Alerting (v0.3.0)
**Goal:** When an alert fires, route it through a time-aware, priority-aware escalation chain so the right person is paged at the right time. Replaces the "single notify_template_key" rule on Phase 13 alert rules with a referenceable, reusable escalation policy.

---

<objective>

## Goal
Layer escalation policies on top of the Phase 13 alert engine. A policy is an ordered list of steps (notify-user, notify-group, notify-oncall, wait, repeat). On-call schedules resolve "who is on duty right now" from a rotation pattern. The escalation worker advances each firing alert through the policy until it acks, resolves, or exhausts the chain.

## Purpose
Phase 13 closed the metric → notification loop but assumes one static recipient. Real ops needs progressive paging: SMS the on-call first, escalate to the lead after 5m of no ack, page the manager after 15m. Without this, every alert wakes the wrong person at 3am.

## Output
- `fct_monitoring_escalation_policies` (id UUID v7, org_id, name, description, is_active, created_at, updated_at, deleted_at)
- `lnk_monitoring_escalation_steps` (policy_id FK, step_order SMALLINT, kind_id FK dim_escalation_step_kind, target_ref JSONB {user_id?|group_id?|schedule_id?}, wait_seconds INT, priority SMALLINT) — immutable per (policy_id, step_order)
- `dim_monitoring_escalation_step_kind` (SMALLINT PK, seeded: notify_user=1, notify_group=2, notify_oncall=3, wait=4, repeat=5)
- `fct_monitoring_oncall_schedules` (id UUID v7, org_id, name, timezone TEXT, rotation_kind_id, rotation_start TIMESTAMPTZ, rotation_period_seconds INT, created_at, updated_at, deleted_at)
- `lnk_monitoring_oncall_members` (schedule_id FK, member_order SMALLINT, user_id FK)
- `dtl_monitoring_alert_escalation_state` (alert_event_id PK, policy_id FK, current_step SMALLINT, next_action_at TIMESTAMPTZ, ack_user_id FK NULL, ack_at NULL, exhausted_at NULL)
- Phase 13 `fct_monitoring_alert_rules.escalation_policy_id` nullable FK added (back-compat: NULL = legacy single-template behavior preserved)
- `escalation_worker.py` runs every 15s: scans `dtl_monitoring_alert_escalation_state` rows where `next_action_at <= now AND ack_at IS NULL AND exhausted_at IS NULL`, resolves the current step's recipient, calls `notify.send.transactional` via NCP v1, advances to next step
- On-call resolver: pure function `resolve_oncall(schedule, at) -> user_id` using rotation index = floor((at - rotation_start) / period) % len(members)
- Nodes: `monitoring.escalation.policy_create/update/delete`, `monitoring.escalation.advance` (effect tx=own), `monitoring.oncall.resolve` (control)
- ≥16 pytest + 1 Robot E2E

</objective>

## AC-2: On-call schedule resolution
```gherkin
Given an on-call schedule with members [alice, bob, carol], rotation_period_seconds=604800 (1w), rotation_start=2026-01-05T00:00:00Z (Mon)
When resolve_oncall(schedule, 2026-04-20T10:00:00Z) is called
Then index = floor((apr20 - jan5) / 604800) % 3 = correct rotating user
And the resolver is timezone-aware via schedule.timezone
And handover boundaries land on the schedule's local midnight, not UTC
And POST /v1/monitoring/oncall-schedules/{id}/whoami returns {user_id, on_until} for now
```

## AC-3: Escalation worker advancement
```gherkin
Given an alert fires with rule.escalation_policy_id set
Then evaluator inserts dtl_monitoring_alert_escalation_state(alert_event_id, policy_id, current_step=0, next_action_at=now)
And escalation_worker (15s tick) processes rows where next_action_at <= now:
  1. Loads step at current_step
  2. Resolves recipient: notify_user → step.target_ref.user_id; notify_group → expand group; notify_oncall → resolve_oncall(target_ref.schedule_id, now)
  3. If kind=wait: just advance current_step, set next_action_at = now + step.wait_seconds
  4. If kind=notify_*: call notify.send.transactional with priority=step.priority, advance current_step, set next_action_at = now + (next step's wait_seconds OR 0 if last)
  5. If kind=repeat: reset current_step to 0
  6. If current_step exceeds steps and no repeat: set exhausted_at=now
And worker commits per-state-row, single failure does not block siblings
```

## AC-4: Ack short-circuits escalation
```gherkin
Given a firing alert in escalation step 2 with next_action_at in 3 minutes
When POST /v1/monitoring/alerts/{id}/ack {note?} is called by an authorized user
Then dtl_monitoring_alert_escalation_state.ack_user_id and ack_at are set
And worker skips this row on subsequent ticks
And alert remains in 'firing' state until rule condition clears (ack ≠ resolve)
And a notify-ack notification is sent to the policy owner (template: alert.acked)
And metric monitoring.escalation.acks_total incremented with labels {policy_id, step}
```

## AC-5: Priority routing through Notify

