# Plan 40-03 — Incident Grouping + Alert Dedup

**Phase:** 40 — Monitoring Alerting (v0.3.0)
**Goal:** Promote firing alerts into incidents — durable, group-by-fingerprint-or-label objects with their own state machine (open → acknowledged → resolved → closed). Dedupe re-fires from the same root cause into a single incident; only escalate the incident, never the individual alert.

---

<objective>

## Goal
Add an incident layer above alerts. The grouper continuously folds firing alert events into incidents using configurable group_by keys (default: rule_id + critical labels). One incident may contain N alert events; closing the incident closes the chain. Escalation operates on incidents, not on raw alerts — collapsing alert-storms into a single page.

## Purpose
Without grouping, 200 hosts breaching the same threshold create 200 escalation chains and 200 pages. Operators see a wall of duplicates instead of one incident saying "200 hosts breached." This plan caps the noise and gives ops a stable handle for triage, status updates, and post-mortems.

## Output
- `fct_monitoring_incidents` (id UUID v7, org_id, group_key TEXT, title TEXT, severity_id FK, state_id FK dim_incident_state, opened_at, acknowledged_at NULL, resolved_at NULL, closed_at NULL, ack_user_id FK NULL, resolved_by_user_id FK NULL, summary TEXT NULL, root_cause TEXT NULL, postmortem_ref TEXT NULL, escalation_state_id FK NULL, action_count INT DEFAULT 0, created_at, updated_at, deleted_at)
- `lnk_monitoring_incident_alerts` (incident_id FK, alert_event_id FK, joined_at, PK (incident_id, alert_event_id)) — immutable lnk
- `dtl_monitoring_incident_grouping_rules` (rule_id FK fct_monitoring_alert_rules PK, group_by JSONB array of label keys, group_window_seconds INT default 300, dedup_strategy TEXT CHECK IN ('fingerprint','label_set','custom_key'), is_active BOOL)
- `dim_monitoring_incident_state` (SMALLINT PK, seeded: open=1, acknowledged=2, resolved=3, closed=4)
- `evt_monitoring_incident_timeline` (id UUID v7, incident_id FK, kind_id SMALLINT FK dim_incident_event_kind, actor_user_id FK NULL, payload JSONB, occurred_at) — append-only audit trail per incident
- `dim_monitoring_incident_event_kind` seeded: created=1, alert_joined=2, acknowledged=3, escalated=4, action_dispatched=5, comment_added=6, resolved=7, closed=8, reopened=9
- Grouper: pure function `compute_group_key(rule, alert_event, grouping_rule) -> str` derived from `dedup_strategy`
- `incident_grouper_worker.py`: subscribes to LISTEN `monitoring_alert_fired` (Phase 13 evaluator emits NOTIFY on transition); resolves group_key, finds open incident with matching (org_id, group_key) within group_window_seconds, links alert OR creates new incident
- State machine: open → acknowledged (via /v1/monitoring/incidents/{id} PATCH state); → resolved (manual OR all linked alerts resolved); → closed (manual after retention period); reopen on new alert join while resolved-not-yet-closed flips back to acknowledged
- Escalation rebind: Plan 40-01 escalation_state.alert_event_id replaced semantically by incident_id (back-compat: column kept, new escalations attach to incident; old per-alert state rows continue draining naturally)
- Action rebind: Plan 40-02 action deliveries gain incident_id FK NULL; rule-attached actions now fire on incident open/resolve transitions (not on every linked alert)
- Nodes: `monitoring.incidents.group` (effect tx=own), `monitoring.incidents.transition` (effect tx=caller), `monitoring.incidents.comment_add` (effect tx=caller)
- ≥20 pytest + 1 Robot E2E

</objective>

## AC-1: Incident creation from first alert
```gherkin
Given a rule with no grouping config (default: dedup_strategy=fingerprint, group_window_seconds=300)
When the rule fires its first alert event
Then the grouper computes group_key = sha256(rule_id || fingerprint)
And no open incident exists for (org_id, group_key) within 300s
And a new fct_monitoring_incidents row is created in state=open with title derived from rule.name + label snippet
And lnk_monitoring_incident_alerts links the alert
And evt_monitoring_incident_timeline records kind=created and kind=alert_joined
```

## AC-2: Dedup of re-fire
```gherkin
Given an incident is open with group_key K
When a second alert event for the same group_key is created within group_window_seconds
Then NO new incident is created
And the existing incident has the new alert linked via lnk_monitoring_incident_alerts
And evt_monitoring_incident_timeline records kind=alert_joined
And metric monitoring.incidents.dedup_total{rule_id} incremented
And no additional escalation is started for this incident (existing escalation_state continues)
```

## AC-3: Custom group_by labels
```gherkin
Given grouping_rule with dedup_strategy=label_set, group_by=["service","env"]
When 50 alerts fire for service=api,env=prod across 50 host labels
Then exactly 1 incident exists with group_key = sha256(rule_id || "service=api|env=prod")
And lnk_monitoring_incident_alerts contains all 50 alert events
And operator pages once, not 50 times
```

## AC-4: State machine
```gherkin
Given an incident in state=open
When PATCH /v1/monitoring/incidents/{id} {state:"acknowledged"} called by user U
Then state_id flips to acknowledged, acknowledged_at and ack_user_id set
And timeline records kind=acknowledged with actor=U
And the linked escalation_state.ack_at is set (short-circuits Plan 40-01 worker)

When all linked alerts have resolved_at set
Then a system transition fires: state → resolved, resolved_at = now, resolved_by_user_id = NULL (system)
And resolve action templates dispatch (per Plan 40-02 hook on transition)

When PATCH state=closed with required {root_cause, summary}
Then state → closed, closed_at set
And no further alert can join this incident (new alerts create a new incident)

When new alert event arrives and finds an incident in state=resolved (within reopen_window=600s)
Then state flips back to acknowledged (auto-reopen), timeline records kind=reopened
```

## AC-5: Escalation rebind to incident
```gherkin

And ack on the incident propagates to escalation_state.ack_at (short-circuits paging)
And legacy rows with alert_event_id-only continue advancing (back-compat read path branches on which FK is set)
```
