<objective>
## Goal
Make the notify Send API safely callable by external systems. Issue scoped API keys stored as argon2id hashes, enforce scopes in middleware, and require an `Idempotency-Key` on `POST /v1/notify/send` so duplicate requests return the same delivery.

## Purpose
Today Send only works with a browser session cookie, so no backend job in another service can call it — blocking every real integration. And a double-click on "send password reset" creates two deliveries. This plan closes both gaps so the notify backend is a production primitive, not a browser-only API.

## Output
- New `03_iam` sub-feature `15_api_keys` with full CRUD (5 files).
- Bearer-token middleware matching the existing session middleware contract.
- `Idempotency-Key` header honored on `POST /v1/notify/send`.
- `/account/api-keys` page: create / copy-once / revoke.
- 2 test files covering key issuance + idempotency.
</objective>

## AC-1: API keys can be created and revoked
```gherkin
Given an authenticated user
When they POST /v1/api-keys with {label, scopes: ["notify:send"], expires_at?}
Then the response body returns the full token exactly once (prefix.secret)
  And the DB stores only the argon2id hash
  And a subsequent GET /v1/api-keys shows the key with masked prefix + scopes + last_used_at
  And DELETE /v1/api-keys/{id} soft-deletes it (revoked_at set)
```

## AC-2: Bearer-token requests authenticate and enforce scopes
```gherkin
Given a valid API key with scope "notify:send"
When a request arrives with Authorization: Bearer nk_<id>.<secret>
Then request.state.user_id and org_id are populated from the key's owner
  And routes decorated as requiring scope "notify:send" succeed
  And routes requiring "audit:read" return 403 FORBIDDEN
  And last_used_at on the key is updated
```

## AC-3: Send API is idempotent
```gherkin
Given a POST /v1/notify/send with header Idempotency-Key: abc123
When the same request is retried within 24h with the same body + key
Then the response is 201 with the same delivery_id as the first call
  And only one row exists in 15_fct_notify_deliveries for that (org_id, idempotency_key) pair
  And the second call does not emit a second audit event
```

## AC-4: Revoked or expired keys are rejected
```gherkin
Given an API key that is revoked_at IS NOT NULL
When a Bearer request arrives using it
Then request.state.user_id remains None (as with any invalid auth)
  And protected endpoints return 401 UNAUTHORIZED
```

## AC-5: /account/api-keys UI works end-to-end
```gherkin
Given an authenticated user on /account/api-keys
When they click "New key", enter a label + pick scopes, and submit
Then the returned token is shown once with a "Copy" button and a warning
  And after reload, the token is gone but the key appears in the list
  And "Revoke" on any key removes it from the list
```

<boundaries>

## DO NOT CHANGE
- Existing session-cookie auth path in `SessionMiddleware.dispatch` — Bearer path is additive.
- `audit.events.emit` node contract or the CHECK constraint on `evt_audit` — API-key events use `audit_category="setup"` like other iam mutations.
- `14_fct_notify_subscriptions` or the worker — subscription-driven deliveries are out of scope for idempotency (they already dedupe on outbox_id).
- Other notify sub-features (06_deliveries repo gets a single new kwarg; no other changes).

## SCOPE LIMITS
- Only two scopes are enforced this plan: `notify:send`, `notify:read`. Adding `audit:read`/`iam:read` etc. is allowed in the dim list but not enforced at endpoints in this plan.
- No rate limiting per key (deferred).
- No per-endpoint scope decorator system — use a single `require_scope` dep used on /v1/notify/send only.
- No admin "view all org keys" page — users see their own keys.

</boundaries>
