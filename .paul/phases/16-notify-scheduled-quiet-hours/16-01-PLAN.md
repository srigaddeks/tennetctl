<objective>
## Goal
Let callers schedule a notification for a future time. `scheduled_at` column already exists on `15_fct_notify_deliveries`; this plan makes sender polls honor it and adds `send_at` / `delay_seconds` inputs to the Send API.

## Purpose
Without scheduled sends you can't build reminders, debounces, drip rules, or any real notification workflow. This is one of the highest-leverage primitives in a notification backend.

## Output
- Email + webpush poll queries filter `(scheduled_at IS NULL OR scheduled_at <= NOW())`.
- `TransactionalSendRequest` gets `send_at: datetime | None` + `delay_seconds: int | None` (mutually exclusive).
- `create_delivery` accepts `scheduled_at`.
- Send UI: "Schedule for" input (optional).
- Test proves delivery with future `scheduled_at` is not picked up, and becomes eligible after time passes.

## Non-goals (deferred)
Quiet hours + per-user timezone requires iam.user schema changes + a preferences UI surface for quiet-window. That ships as v0.1.8 plan 16-02 once we touch iam user profile.
</objective>
