<objective>
## Goal
Ship the Vault frontend end-to-end: top-bar feature registration, `/vault` list page with metadata table, create-secret dialog that reveals the value exactly once after write, rotate dialog with one-time reveal of the new value, delete-with-confirm, TanStack Query hooks with cache-aware mutations, and a Robot Framework + Playwright Browser E2E suite proving the full CRUD flow against the live backend. After this plan, vault is fully usable by an operator via the browser.

## Purpose
Plan 07-01 ships the backend only; operators still need to reach the vault through `curl`. 07-02 makes vault a first-class UI feature visible from the top bar, matching the Feature Flags + IAM UIs already shipped. Reveal-once dialog enforces the "values never re-displayed" commitment from ADR-028 in the UI itself — users cannot accidentally leave a plaintext secret on-screen after navigation.

## Output
- `Vault` entry added to `frontend/src/config/features.ts` — top-bar tab + sidebar sub-features (`All secrets` + future `Config` slot reserved)
- `frontend/src/features/vault/` package: hooks (`use-secrets.ts`) + `_components/` for all dialogs
- `/vault` page: list table with key / version / description / rotated_at / actions; filter input; empty-state CTA
- Reveal-once dialog: shows the plaintext exactly once, with a copy-to-clipboard button, a 3-second warning banner, and a single dismiss that unmounts the value from React state
- Three mutations wired through TanStack Query with optimistic list invalidation: create, rotate, delete
- Robot E2E suite under `tests/e2e/vault/01_secrets.robot`: runs headless Chromium via Playwright Browser library, drives the list → create → rotate → delete flow, asserts reveal-once appears and that the value never reappears in list/detail views afterwards

## Reveal-once policy (UI-level enforcement of ADR-028)
- After `POST /v1/vault` succeeds, the response does NOT contain the value — so the create dialog holds the user-entered value in a ref and, on successful mutation, hands that ref to the reveal-once dialog, then clears the ref.
- Reveal-once dialog: monospace textarea, copy button, "This value will not be shown again. Dismiss when stored." warning, single "Got it" button that unmounts + garbage-collects the string.
- Rotate dialog: same pattern — on success, show the new value once.
- No route anywhere uses `GET /v1/vault/{key}` (which returns plaintext) except the explicit backend-internal path — the UI does not need to re-read values.

    **`features/vault/hooks/use-secrets.ts`** — TanStack hooks, mirror `use-flags.ts` shape:
    ```typescript
    "use client";
    import { useMutation, useQuery, useQueryClient, type UseQueryResult } from "@tanstack/react-query";
    import { apiFetch, apiList } from "@/lib/api";
    import type { ListResult, VaultSecretMeta, VaultSecretCreateBody, VaultSecretRotateBody } from "@/types/api";

    const key = {
      all: ["vault", "secrets"] as const,
      list: () => ["vault", "secrets", "list"] as const,
    };

    export function useSecrets(): UseQueryResult<ListResult<VaultSecretMeta>> {
      return useQuery({
        queryKey: key.list(),
        queryFn: () => apiList<VaultSecretMeta>("/v1/vault"),
      });
    }

    export function useCreateSecret() {
      const qc = useQueryClient();
      return useMutation({
        mutationFn: (body: VaultSecretCreateBody) =>
          apiFetch<VaultSecretMeta>("/v1/vault", { method: "POST", body: JSON.stringify(body) }),
        onSuccess: () => qc.invalidateQueries({ queryKey: key.all }),
      });
    }

    export function useRotateSecret(secretKey: string) {
      const qc = useQueryClient();
      return useMutation({
        mutationFn: (body: VaultSecretRotateBody) =>
          apiFetch<VaultSecretMeta>(`/v1/vault/${encodeURIComponent(secretKey)}/rotate`, {
            method: "POST",
            body: JSON.stringify(body),
          }),
        onSuccess: () => qc.invalidateQueries({ queryKey: key.all }),
      });
    }

    export function useDeleteSecret(secretKey: string) {
      const qc = useQueryClient();
      return useMutation({
        mutationFn: () =>
          apiFetch<void>(`/v1/vault/${encodeURIComponent(secretKey)}`, { method: "DELETE" }),
        onSuccess: () => qc.invalidateQueries({ queryKey: key.all }),
      });
    }
    ```

    Critical: the mutation hooks do NOT cache the body arg or the response value. They return the metadata only; the raw plaintext user entered stays in the dialog's ref.

    Avoid:
    - Storing the raw plaintext in React Query cache or any global state
    - Using `any` — types are defined
    - Including the secret key in the queryKey for create/rotate mutations (not a query key — it's a mutation arg)

    ```typescript
    "use client";
    import { useRef, useState } from "react";
    import { useForm } from "react-hook-form";
    import { zodResolver } from "@hookform/resolvers/zod";
    import { z } from "zod";
    import { useCreateSecret } from "@/features/vault/hooks/use-secrets";
    import { Modal } from "@/components/modal";
    import { RevealOnceDialog } from "./reveal-once-dialog";
    import { useToast } from "@/components/toast";

    const KEY_RE = /^[a-z][a-z0-9._-]{0,127}$/;

    const schema = z.object({
      key: z.string().regex(KEY_RE, "lowercase letter, digit, dot, dash, underscore"),
      value: z.string().min(1, "value is required").max(65536),
      description: z.string().max(500).optional(),
    });

    type Form = z.infer<typeof schema>;

    export function CreateSecretDialog({ open, onClose }: { open: boolean; onClose: () => void }) {
      const revealRef = useRef<string | null>(null);
      const [revealOpen, setRevealOpen] = useState(false);
      const { register, handleSubmit, reset, formState: { errors, isValid } } = useForm<Form>({
        resolver: zodResolver(schema),
        mode: "onChange",
      });
      const create = useCreateSecret();
      const toast = useToast();

      const onSubmit = handleSubmit(async (data) => {
        try {
          await create.mutateAsync({ key: data.key, value: data.value, description: data.description ?? null });
          revealRef.current = data.value;
          onClose();
          reset();
          setRevealOpen(true);
          toast.success(`Created secret ${data.key}`);
        } catch (e) {
          // ApiClientError 409 shown inline via create.error
        }
      });

      return (
        <>
          <Modal open={open} onClose={onClose} title="New secret">
            <form onSubmit={onSubmit} data-testid="create-secret-form">
              {/* key / value / description fields with data-testids and error helpers */}
              {/* Inline server-error: {create.error ? <div className="text-red-600">{create.error.message}</div> : null} */}
            </form>
          </Modal>
          <RevealOnceDialog
            open={revealOpen}
            value={revealRef.current ?? ""}
            onDismiss={() => {
              revealRef.current = null;
              setRevealOpen(false);
            }}
          />
        </>
      );
    }
    ```

    Test ids: `new-secret-key`, `new-secret-value`, `new-secret-description`, `new-secret-submit`, `new-secret-error`.

    Avoid:
    - Putting the value in component state that persists after dismiss
    - Leaving the form's `value` field populated after success (reset clears it)
    - Logging the value anywhere (no `console.log`)
  </action>

<considerations>

## Why reveal-once is UI-level (not just backend)

Backend never returns the value in list responses — that's ADR-028. But without the UI enforcement:
- A create dialog could re-render the just-submitted value after network delay.
- React Query might cache the POST body in a mutation's `data`.
- A paste into the clipboard could re-surface.

The reveal-once dialog solves all three by holding the value in a `useRef` (not state), showing it once in an unmounting component, and clearing the ref on dismiss. Test "Reveal Once Is Truly Once" asserts a page reload also cannot recover it — belt-and-suspenders.

## Test ID scheme

Every interactive element has a deterministic `data-testid`. Row-scoped ones include the key: `secret-row-${key}`, `secret-row-rotate-${key}`, `secret-row-delete-${key}`. This lets Robot drive without CSS/positional selectors and makes the tests stable across UI restyling.

## Shared Robot resources

`keywords/api.resource` + `keywords/vault.resource` are pulled out because plan 08 auth E2E will reuse `API Delete Vault Key` (for cleaning up signing keys in test tenants) and `Open Vault Page` (for screenshotting signing-key rotation). Creating them now as a side-benefit of this plan.

## Known gaps (for SUMMARY, not blocking)

- No "paginated list" — v0.2 assumes <100 secrets per deployment. Pagination UI and `limit/offset` params arrive if a deployment hits the cap.
- No secret-history view — old versions are retained 7 days and purged; no UI surface yet.
- No copy-to-clipboard fallback for browsers without `navigator.clipboard` (insecure context). Acceptable: vault admin page is only meaningful over HTTPS in prod; dev uses localhost which is considered secure.
- No per-secret audit trail UI. Audit events are written by the backend, but the Audit viewer is a separate (future) feature. A `View audit` link from each secret row is pre-wired as a future TODO comment, not implemented.

</considerations>
