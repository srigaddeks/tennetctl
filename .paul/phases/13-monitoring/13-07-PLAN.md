# Plan 13-07 — Retention, Rollups, Partition Management, Synthetic Checks, LISTEN/NOTIFY wake-up

**Phase:** 13 — Monitoring
**Goal:** Make storage sustainable. Wire `pg_cron` to drive 1m/5m/1h rollups, daily-partition create/drop with hot/warm/cold tiering, add synthetic HTTP checks (blackbox replacement), switch log live-tail from polling to LISTEN/NOTIFY for real-time.

---

<objective>

## Goal
Close long-running operational concerns: rollup aggregation, partition lifecycle, ingest latency → UI, and dead-man's switch checks against external endpoints.

## Purpose
Without rollups, queries over 30 days scan billions of rows. Without partition management, the DB fills up and INSERTs stall. Without synthetic checks, "is my service up?" can't be answered when the service itself is producing no logs. LISTEN/NOTIFY replaces 1s-poll live-tail with sub-100ms delivery.

## Output
- `pg_cron` enabled in Postgres container + 3 scheduled jobs (rollup_1m every 1m, rollup_5m every 5m, rollup_1h every hour) + 1 daily partition-manager job
- Idempotent PL/pgSQL rollup procs: upsert aggregates into rollup tables from raw evt_monitoring_metric_points within [last_bucket, now]
- PL/pgSQL partition manager: creates partitions 7 days ahead, drops partitions per retention policy (hot=7d raw, warm=30d 1m rollup, cold=90d 1h rollup)
- Retention policy table: `fct_monitoring_retention_policies` (per table: raw/rollup, days_to_keep, tier)
- Synthetic check sub-feature: CRUD + runner worker that hits configured HTTP targets every N seconds, records latency histogram + up/down gauge
- Log LISTEN/NOTIFY: insert trigger on evt_monitoring_logs publishes to channel `monitoring_logs_new` (not per-row — per-batch aggregate notify). Live-tail endpoint subscribes.
- Notify listener: consumes Postgres NOTIFY for low-latency UI push
- ≥15 pytest green

</objective>

## AC-2: Rollup correctness
```gherkin
Given raw points exist for metric_id=1 across 10 distinct minute buckets
When monitoring_rollup_1m runs
Then evt_monitoring_metric_points_1m has 10 rows per (metric_id, labels_hash, resource_id)
And each row has correct count, sum, min, max, last
And re-running the proc is idempotent (ON CONFLICT DO UPDATE with latest values)
And histograms roll up by element-wise sum of histogram_counts arrays
And gauges use `last` (most recent value in bucket), not sum
```

## AC-3: Partition lifecycle
```gherkin
Given monitoring_partition_manager runs daily
Then for each of (logs, spans, metric_points, metric_points_1m, metric_points_5m, metric_points_1h):
  - Creates partitions 7 days into the future if missing
  - Drops partitions older than retention policy for that table
And policies from fct_monitoring_retention_policies drive the decisions:
  - evt_monitoring_logs: 14 days
  - evt_monitoring_spans: 7 days
  - evt_monitoring_metric_points (raw): 7 days
  - evt_monitoring_metric_points_1m: 30 days
  - evt_monitoring_metric_points_5m: 90 days
  - evt_monitoring_metric_points_1h: 365 days
And policy changes via UPDATE on fct_monitoring_retention_policies take effect next daily run
```

## AC-4: Synthetic checks
```gherkin
Given fct_monitoring_synthetic_checks (id, org_id, name, target_url, method, expected_status, timeout_ms, interval_seconds, headers JSONB, body TEXT, assertions JSONB, is_active, created_at, updated_at) exists
When a check is active and interval elapsed
Then synthetic_runner worker:
  - HTTP GET (or specified method) the target
  - Records: monitoring.synthetic.up (gauge 1/0), monitoring.synthetic.duration_ms (histogram)
  - Labels include {check_name, target_host}
  - On failure 3× consecutive: emits monitoring.synthetic.down audit event (category=system)
  - CRUD endpoints: GET list, POST create, GET one, PATCH, DELETE soft-delete
```

## AC-5: LISTEN/NOTIFY for log tail
```gherkin
Given an INSERT trigger on evt_monitoring_logs (AFTER INSERT)
Then the trigger executes pg_notify('monitoring_logs_new', row_to_json(minimal_payload))
And notify_listener worker holds a dedicated asyncpg connection, LISTENs to the channel
And fans notifications out to subscribed SSE clients via an asyncio.Queue broadcaster
And live-tail endpoint uses this broadcaster instead of 1s polling
And the trigger payload is truncated to 3KB (asyncpg NOTIFY limit is ~8KB minus overhead; err on safe side) — if body too long, payload includes id + time only, client fetches full row
```
