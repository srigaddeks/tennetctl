# Plan 13-04 — JetStream Consumers + Log Redaction + APISIX Scraper

**Phase:** 13 — Monitoring
**Goal:** Drain `MONITORING_LOGS` and `MONITORING_SPANS` streams into Postgres via the storage seam. Apply a configurable redaction pipeline to logs before persistence. Add a pull-mode APISIX Prometheus scraper that converts APISIX's `/apisix/prometheus/metrics` text output into `MetricsStore` writes.

---

<objective>

## Goal
Close the ingest loop: whatever OTel clients push (13-03) or the backend emits (instrumentation) lands in Postgres via idempotent, durable consumers. Plus an APISIX sidecar that gives us gateway observability for free.

## Purpose
Durability comes from JetStream; correctness comes from transactional batch inserts; security comes from redaction. The APISIX scraper is leverage — the gateway already emits Prometheus metrics on `:51791`, no reason to ignore it.

## Output
- 2 durable JetStream consumers running as asyncio tasks in the FastAPI lifespan
- Batch size 200 / flush interval 1s / max deliver 5 / ack-wait 30s
- Dead-letter subject `monitoring.dlq.logs` + `monitoring.dlq.spans` (stream `MONITORING_DLQ` added to bootstrap)
- Redaction engine with regex + attribute-key denylist + vault-controlled rules
- APISIX scraper polling every 15s, parsing Prom text format → counter/gauge/histogram writes
- ≥15 pytest green

</objective>

## AC-1: Logs consumer durability
```gherkin
Given MONITORING_LOGS stream has messages
When logs_consumer runs
Then it pulls batches of ≤200 messages (or 1s timeout)
And decodes each OTLP ResourceLogs batch to LogRecord list
And calls ResourcesStore.upsert per unique resource (intern)
And calls LogsStore.insert_batch in one transaction
And acks all messages on success
And nacks (redeliver) on transient errors up to max_deliver=5
And redelivery exhausted → publishes message to monitoring.dlq.logs and acks original
```

## AC-2: Spans consumer
```gherkin
Given MONITORING_SPANS stream has messages
Then the consumer drains to PostgresSpansStore with identical durability semantics
And trace_id/span_id hex ↔ bytes conversion handled correctly
And duration_ns is computed by the generated column from start/end
```

## AC-3: Redaction
```gherkin
Given fct_monitoring_redaction_rules has rows (pattern, applies_to: body|attribute|both, kind: regex|denylist, replacement)
When the logs consumer processes a LogRecord
Then each rule is applied in order:
  - regex rules on body + attribute string values
  - denylist rules drop entire attribute if key matches (default: password, token, authorization, cookie, set-cookie, api_key, secret, ssn, credit_card)
And dropped_attributes_count is incremented when attributes are redacted away
And default rules are seeded: credit card (13-19 digits pattern), email (optional — off by default), AWS access keys (AKIA*), JWT tokens (eyJ* prefix), bearer tokens
And rule changes take effect within 60s (cache TTL)
```

## AC-4: APISIX scraper
```gherkin
Given APISIX is reachable on 51791
When the scraper runs every 15s
Then it GETs http://apisix:9091/apisix/prometheus/metrics
And parses Prometheus text format (counter, gauge, histogram)
And auto-registers metrics on first scrape (prefixed "apisix.*")
And writes points to evt_monitoring_metric_points
And uses labels_hash interning where possible
And on scrape failure: logs warning, emits monitoring.scrapers.failures_total counter, continues next cycle
And the scraper is gated by config.MONITORING_APISIX_SCRAPE_ENABLED (default true)
```

<tasks>

## Task 1 — Redaction rules migration (040)
`20260417_040_monitoring-redaction-rules.sql`:
- `fct_monitoring_redaction_rules` (id SMALLINT IDENTITY PK, org_id UUID NULL means global, code TEXT UNIQUE, pattern TEXT, applies_to TEXT CHECK IN ('body','attribute','both'), kind TEXT CHECK IN ('regex','denylist'), replacement TEXT DEFAULT '[REDACTED]', is_active BOOL DEFAULT true, priority SMALLINT DEFAULT 100, created_at, updated_at)
- `v_monitoring_redaction_rules` view

## Task 2 — Redaction seed
`05monitoring_06_redaction_rules.yaml` — default rules (credit card, AWS key, JWT, bearer, password/token/cookie denylist).

## Task 3 — Redaction engine
`workers/redaction.py`:
- `RedactionEngine.load(pool)` — caches rules, refreshes every 60s
- `apply(record: LogRecord) -> LogRecord` — returns new record (immutability rule) with body and attributes redacted; increments dropped_attributes_count
- Compiled regex cache
- Unit-testable in isolation (no DB required once loaded)

## Task 4 — Logs consumer
`workers/logs_consumer.py`:
- Pull-subscribe to MONITORING_LOGS with durable name "monitoring-logs-postgres"
- Loop: `fetch(batch=200, timeout=1s)` → decode OTLP → intern resources → redact → batch insert → ack
- On error: nack all (individual failures unlikely since batch is a single tx)
- DLQ: check delivery count from message metadata; if >= max_deliver-1, publish to DLQ subject before ack

## Task 5 — Spans consumer
`workers/spans_consumer.py` — parallel to logs consumer; no redaction (spans typically don't carry PII body; defer redaction to v0.2 if needed)

## Task 6 — APISIX scraper
`workers/apisix_scraper.py`:
- Use `httpx.AsyncClient` to GET APISIX metrics
- Parse Prometheus text format using `prometheus-client` library exposition parser (small dep, or hand-roll 60-line parser)
- Convert to MetricsStore writes; register once per unique metric name
- Attribute every metric with label `source=apisix`
- Use single DB connection; long-lived

## Task 7 — Worker supervisor
`workers/runner.py`:
- `WorkerPool` class with `start(pool, nats_js, config)` / `stop()` / `health()` async methods
- Supervises 3 workers
- Exponential backoff on crash
- Tracks last heartbeat per worker; `health()` returns dict of worker_name → {running, last_heartbeat, restart_count}

## Task 8 — Extend JetStream bootstrap
Add MONITORING_DLQ stream to `bootstrap/jetstream.py`.

## Task 9 — Lifespan wiring
`backend/main.py`:
```python
worker_pool: WorkerPool | None = None
...
if config.MONITORING_ENABLED and nats_connected:
    worker_pool = WorkerPool()

...
# shutdown
if worker_pool:
    await worker_pool.stop(timeout=10)
```
