# Plan 13-08 — Alerting Engine + Notify Integration + Silences + UI

**Phase:** 13 — Monitoring (final plan)
**Goal:** Close the observability loop: from metric or log pattern → rule → firing alert → delivered notification. Integrates with the existing Notify feature (Phase 11) to route via the `critical` category for multi-channel fan-out.

---

<objective>

## Goal
Replace Prometheus Alertmanager. Rules run on a schedule, evaluate DSL queries, fire alerts when thresholds breach for longer than a `for_duration`, auto-resolve when the condition clears, and deduplicate via fingerprints. Silences suppress alerts for planned windows. Notify integration means alerts reach email/webpush/in-app without reinventing delivery.

## Purpose
Telemetry without paging is passive. This plan makes the system actively wake people up when things go wrong — and only when things go wrong.

## Output
- `fct_monitoring_alert_rules` (id, org_id, name, description, target: metrics|logs, dsl JSONB, condition: {op, threshold, for_duration}, severity_id FK, notify_template_key TEXT, labels JSONB, is_active, paused_until, created_at, updated_at)
- `evt_monitoring_alert_events` (id UUID v7, rule_id FK, fingerprint TEXT, state CHECK IN ('firing','resolved'), value DOUBLE PRECISION, started_at, resolved_at NULL, last_notified_at, notification_count INT, labels JSONB, annotations JSONB, PK (id))
- `fct_monitoring_silences` (id, org_id, matcher JSONB, starts_at, ends_at, reason TEXT, created_by)
- Evaluator worker: runs every 30s (configurable), loads active rules, runs each rule's DSL, evaluates condition, creates/updates evt_monitoring_alert_events, emits Notify transactional sends for firing/resolved transitions
- Fingerprint = hash(rule_id + label_set) — enables auto-resolve and dedup
- Silence matching evaluated at firing time: silenced alerts are recorded but not notified
- `monitoring:admin` scope gates rule mutations; `monitoring:write` gates silences
- ADR-reference: lean on Phase 11 Notify for delivery — no new delivery code here
- ≥18 pytest + 1 Robot E2E

</objective>

## AC-1: Rule CRUD
```gherkin
Given POST /v1/monitoring/alert-rules with {name, target, dsl, condition:{op:gt,threshold:0.05,for_duration:"5m"}, severity, notify_template_key, labels?}
Then a row is created in fct_monitoring_alert_rules
And nodes exist: monitoring.alerts.rule_create/update/delete (effect+audit), get/list (control)
And dsl is validated against 13-05 DSL schema
And notify_template_key must resolve to an existing notify template or reject 400
```

## AC-2: Evaluator lifecycle
```gherkin
Given alert_evaluator_worker runs every 30s
Then for each active rule:
  1. Compile + run rule.dsl via 13-05 compiler with implicit timerange = [now - for_duration, now]
  2. For each row in the result (grouped by labels):
     - Compute value, check condition (gt/gte/lt/lte/eq/ne)
     - If breached: compute fingerprint = sha256(rule_id || sorted(labels))
     - Upsert evt_monitoring_alert_events: if existing row with same fingerprint and state='firing' → update value + last_seen; else insert new with state='firing', started_at=now
  3. For fingerprints not present in current result but state='firing' → flip to 'resolved', set resolved_at=now
And evaluator commits per-rule (single failing rule doesn't break others)
And metric monitoring.alerts.evaluations_total + duration histogram + rules_active gauge emitted
```

## AC-3: For-duration gating
```gherkin
Given a rule with for_duration=5m
When the condition breaches at t0
Then no alert event is emitted yet (value tracked internally)
When the condition remains breached at t0+5m
Then alert event firing is recorded AND notification sent
When the condition clears before 5m elapsed
Then no alert event is ever created (transient blip suppressed)
```

## AC-4: Notify integration
```gherkin
Given alert transitions to firing
Then evaluator calls POST /v1/notify/send (or notify.send.transactional node) with:
  - template_key = rule.notify_template_key
  - recipient: resolved from rule.labels.user_id or default org on-call list
  - channel: per template's default OR critical if rule.severity=critical (fan-out)
  - variables: {rule_name, value, threshold, labels, alert_url: /monitoring/alerts/{id}}
And when alert transitions to resolved
Then a resolved-notification is sent with the same template (template checks {{state}} variable to render the correct body)
And last_notified_at + notification_count tracked on evt_monitoring_alert_events
And re-notification throttle: no more than 1 notification per fingerprint per 15 minutes even if condition re-fires
```

## AC-5: Silences
```gherkin
Given POST /v1/monitoring/silences with {matcher: {rule_id?, labels?: {...}}, starts_at, ends_at, reason}
Then silence row created
And during the silence window:
  - Matching alerts STILL fire in evt_monitoring_alert_events (audit trail intact)
  - But notification is suppressed + alert annotated silenced=true
And expired silences are auto-cleaned (ends_at < now hidden from list, retained in DB)
```

## AC-6: UI
```gherkin
Given user navigates to /monitoring/alerts
Then 4 pages exist:

  - /alerts/rules: rule list
  - /alerts/rules/[id]: rule editor — DSL builder, condition form, notify template picker, test-evaluate button (runs dsl against current data, shows would-fire preview)
```
