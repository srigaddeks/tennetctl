# Plan 13-02 — Metrics Ingest (Counter + Gauge + Histogram)

**Phase:** 13 — Monitoring
**Goal:** Ship the complete metrics ingest path: registry CRUD, 4 catalog nodes (`monitoring.metrics.register/increment/set/observe`), HTTP API (`POST /v1/monitoring/metrics/*`), in-process SDK (`counter`, `gauge`, `histogram` factories), cardinality enforcement with audit-of-reject, and pytest coverage. No NATS involvement — metrics write direct to Postgres via `MetricsStore`.

---

<objective>

## Goal
Make metrics usable from any backend code with one import (`from backend.02_features.05_monitoring.sdk import metrics`) and from any HTTP client via REST. Counter, gauge, histogram all work end-to-end with cardinality caps.

## Purpose
Metrics ingest is synchronous and cheap (single Postgres write). No need for NATS. The SDK layer means developers never see the storage seam — they call `metrics.counter("auth.signin.attempts").increment(labels={...})` and the SDK handles registry lookup, caching, and insertion.

## Output
- 4 catalog effect nodes (all emit audit on register/reject; skip audit on hot-path increment/set/observe per vault precedent)
- 4 REST endpoints: `POST /v1/monitoring/metrics` (register), `POST /v1/monitoring/metrics/{key}/increment`, `POST /v1/monitoring/metrics/{key}/set`, `POST /v1/monitoring/metrics/{key}/observe`
- SDK: in-process metric handle cache, auto-register on first use, zero-alloc hot path
- Cardinality reject audit event: `monitoring.metrics.cardinality_exceeded` (category=system, outcome=failure bypass for scope)
- ≥18 pytest tests green

</objective>

## AC-1: Register a metric
```gherkin
Given an authenticated request with body {key, kind, label_keys, description?, unit?, histogram_buckets?, max_cardinality?}
When POST /v1/monitoring/metrics is called
Then a row exists in fct_monitoring_metrics
And kind=histogram requires non-empty histogram_buckets (400 otherwise)
And kind in {counter,gauge} must NOT provide histogram_buckets (400 otherwise)
And re-posting same key is idempotent (200 returns existing id; description/unit update if changed)
And audit event monitoring.metrics.registered emitted
```

## AC-2: Counter increment
```gherkin
Given a registered counter metric
When POST /v1/monitoring/metrics/{key}/increment with {labels?: {}, value?: 1, resource?: {service_name,...}}
Then a row is inserted into evt_monitoring_metric_points with value=body.value (default 1)
And resource_id is resolved via ResourcesStore.upsert (interned, new row only on hash miss)
And value < 0 for counter returns 400 (counters monotonic)
And labels outside registered label_keys returns 400
And cardinality limit exceeded → 429 + audit monitoring.metrics.cardinality_exceeded (no point written)
```

## AC-3: Gauge set
```gherkin
Given a registered gauge metric
When POST /v1/monitoring/metrics/{key}/set with {labels, value}
Then a row is inserted into evt_monitoring_metric_points with value=body.value (negative allowed)
And MetricsStore.query_latest(metric_id, labels) returns the most recent value
```

```gherkin
Then evt_monitoring_metric_points row has histogram_counts array (Le-bucketed cumulative counts, length = buckets+1 for +Inf), histogram_sum=0.042, histogram_count=1
And 10 observations of varying values produce expected cumulative bucket counts
```

```gherkin

And concurrent first-use is safe (asyncio.Lock around register)
```

## Task 1 — Pydantic schemas
`schemas.py`:
- `MetricRegisterRequest` (key, kind: Literal["counter","gauge","histogram"], label_keys: list[str], description, unit, histogram_buckets: list[float] | None, max_cardinality: int = 1000)
- `MetricIncrementRequest` (labels: dict[str,str] = {}, value: float = 1.0, resource: ResourceIdentity | None)
- `MetricSetRequest` / `MetricObserveRequest`
- `ResourceIdentity` (service_name, service_instance_id, service_version, attributes)
- `MetricResponse`, `MetricListResponse`

## Task 2 — Repository
`repository.py` thin wrapper calling `get_metrics_store(pool)` and `get_resources_store(pool)`. Pass `conn`, not `pool`, to store methods.

## Task 3 — Service
`service.py` business logic:
- `register_metric(pool, conn, ctx, req)` → validates kind+buckets pairing, calls store.register, emits `monitoring.metrics.registered` via `run_node("audit.events.emit", ...)`
- `increment(pool, conn, ctx, key, req)` → resolve metric_id (cache 60s), resolve resource_id, check label_keys match, delegate to store; on cardinality False → emit `monitoring.metrics.cardinality_exceeded` + raise CardinalityError → 429
- `set_gauge` / `observe_histogram` parallel
- Label validation: keys must be subset of registered `label_keys`; all values are TEXT

## Task 4 — Routes
`routes.py`: 4 endpoints + `GET /v1/monitoring/metrics` (list) + `GET /v1/monitoring/metrics/{key}` (one). All use `{"ok": ..., "data": ...}` envelope.

## Task 5 — Nodes
4 effect node handlers in `nodes/`:
- `register.py`: emits audit (register is not hot-path)
- `increment.py`, `set_gauge.py`, `observe.py`: hot-path — skip audit emission, document in manifest the `hot_path: true` marker. Per 13-01 + catalog rules, mark in manifest but still register as effect (writes rows). Use manifest convention established in vault for read-path bypass.

## Task 6 — Manifest update
Extend `feature.manifest.yaml` sub-feature `metrics`:
- 4 nodes (kind=effect each; tx=caller; register has emits_audit=true, others hot_path=true)
- 6 routes (register POST, list GET, one GET, increment POST, set POST, observe POST)

## Task 7 — SDK
`sdk/metrics.py`:
```python
class CounterHandle:
    def __init__(self, key, label_keys, description, unit, max_cardinality): ...
    async def increment(self, ctx, *, labels=None, value=1.0): ...  # registers on first use
def counter(key, *, labels, description=None, unit="1", max_cardinality=1000) -> CounterHandle: ...
def gauge(...) -> GaugeHandle: ...
def histogram(..., buckets) -> HistogramHandle: ...
```
- Module-level `_registered: dict[str, int]` cache + `_locks: dict[str, asyncio.Lock]`
- All calls via `run_node` (never direct store access)
- Respect `config.MONITORING_ENABLED` — no-op if false
