# Plan 13-05 — Query API + Query DSL + Saved Queries + DLQ Admin

**Phase:** 13 — Monitoring
**Goal:** Make the stored data queryable. Ship ADR-028 (monitoring query DSL — JSON-based, safelisted), implement the compiler + validator, expose query endpoints for logs / metrics / traces, add saved queries sub-feature, add admin DLQ replay route.

---

<objective>

## Goal
One DSL compiles to SQL against `v_monitoring_*` views. Callers send a JSON query; the compiler validates against a schema, parameterizes every filter, and returns results with a cursor. UI, alerts, and external integrations all use the same DSL.

## Purpose
Users have metrics/logs/traces in Postgres but no way to ask questions. Rather than invent PromQL, we stick with tennetctl's "SQL-first + view-based" posture and give the DSL a thin skin so it's still JSON-serializable and safe.

## Output
- **ADR-028**: Monitoring Query DSL decision
- Query DSL with 3 shapes: `logs`, `metrics`, `traces`; shared `filter`, `timerange`, `cursor` primitives
- Compiler: DSL → parameterized asyncpg SQL (NEVER string-concat for values)
- Validator: schema-check + SQL-injection-safe + timerange-cap (max 90 days)
- 7 new REST endpoints:
  - `POST /v1/monitoring/logs/query` — log search
  - `POST /v1/monitoring/metrics/query` — metrics timeseries
  - `POST /v1/monitoring/traces/query` — trace search
  - `GET /v1/monitoring/traces/{trace_id}` — single trace waterfall
  - `GET/POST/PATCH/DELETE /v1/monitoring/saved-queries` — saved query CRUD
- 3 catalog query nodes (kind=control, tx=caller, read-only)
- 1 admin route `POST /v1/monitoring/dlq/replay` (scope-gated)
- `GET /health/monitoring` worker-pool health endpoint
- ≥20 pytest green

</objective>

## AC-2: DSL shape
```gherkin
Given the DSL reference
Then every query has:
  - target: "logs" | "metrics" | "traces"
  - filter: tree of {and:[], or:[], not:{}, eq, ne, in, nin, lt, lte, gt, gte, contains, jsonb_path, regex_limited}
  - timerange: {from, to} absolute OR {last: "1h" | "24h" | "7d" | "30d"}
  - limit: int (max 1000)
  - cursor: opaque base64 token for keyset pagination
For metrics:
  - metric_key: required
  - labels: dict filter
  - aggregate: "sum" | "avg" | "min" | "max" | "count" | "rate" | "p50" | "p95" | "p99" (p* only on histograms)
  - bucket: "1m" | "5m" | "1h" | "1d" (auto-selects rollup table)
  - groupby: list[str] (label names)
For logs:
  - severity_min: SMALLINT
  - body_contains: TEXT
  - trace_id: TEXT optional
For traces:
  - service_name, span_name, duration_min_ms, duration_max_ms, has_error: bool, trace_id
  - Single-trace mode: GET /traces/{trace_id} returns tree-structured spans
```

## AC-3: Compiler + safety
```gherkin
Given any DSL query
When the compiler runs
Then output is (sql: str, params: list[Any])
And every user-supplied value appears ONLY in params (zero f-string interpolation)
And SQL whitelist: only SELECT against v_monitoring_* views, no JOIN except what the compiler itself generates
And timerange has a hard cap: to - from ≤ 90 days, reject otherwise
And regex_limited caps pattern length to 100 chars + disallows nested quantifiers (ReDoS protection)
And org_id filter is ALWAYS injected from NodeContext — never from user input
And workspace_id likewise when present in context
```

## AC-4: Metrics query picks rollup
```gherkin
Given a metrics query with bucket=1h and timerange=30d
When the compiler runs
Then it selects from evt_monitoring_metric_points_1h (not raw evt_monitoring_metric_points — which would scan billions of rows)
And for bucket=1m it selects from _1m
And for a range ≤1h with bucket=1m, compiler may select from raw table (finer granularity available)
And aggregate p95 over a histogram uses histogram_counts percentile approximation
```

## AC-5: Endpoints return ok envelope
```gherkin
All 7 endpoints return {"ok": true, "data": {results, cursor?}} on success
Authentication required; cross-org queries return 403 per audit-analytics pattern
And cursor round-trips correctly (follow to exhaustion, all rows returned exactly once)
```
