# Plan 13-01 — Monitoring Foundation: Schema + Storage Seam + JetStream Bootstrap

**Phase:** 13 — Monitoring (Postgres-backed, ClickHouse-ready)
**Milestone:** v0.1.5 Observability
**Goal:** Lay the full monitoring foundation: OTel-aligned schemas for logs + spans + metrics (counter/gauge/histogram) + resource intern table + rollup tables, pluggable `LogsStore`/`MetricsStore`/`SpansStore`/`ResourcesStore` Protocols (Postgres impls first, ClickHouse later), NATS JetStream streams for logs + traces, feature registered in catalog. No ingest endpoints, no UI, no query API yet — that's 13-02 through 13-08.

---

<objective>

## Goal
Build the complete monitoring data model and storage seam in one plan so nothing downstream needs schema changes. Covers all three OTel pillars (logs, metrics, traces), both mutable state (gauges) and observational state (histograms), rollup tables for long-range queries, resource interning for cardinality control, and JetStream stream bootstrap for durable ingest.

## Purpose
Replace Prometheus/Loki/Tempo/Grafana with a single self-hosted observability layer. Ingest path uses NATS JetStream (durable, backpressure-safe) for OTel logs + spans; metrics write direct to Postgres with cardinality limits. Storage seam means ClickHouse is a drop-in swap in v0.2 via `TENNETCTL_MONITORING_STORE_KIND=clickhouse` — no caller code changes.

## Output
- `05_monitoring` schema: 4 dim + 2 fct (metric registry, resource intern) + 3 evt (logs, metric points, spans) + 3 rollup (1m/5m/1h metric aggregates) + 4 views, all event/rollup tables daily range-partitioned on `recorded_at`
- Feature registered in catalog with 3 empty sub-features (logs, metrics, traces)
- 4 store Protocols + 4 Postgres implementations behind `get_*_store()` factories
- `MONITORING_LOGS` + `MONITORING_SPANS` JetStream streams created idempotently on boot
- `nats-py`, `opentelemetry-proto` added to pyproject
- Pytest: all store round-trips + JetStream idempotent bootstrap

</objective>

## Task 12 — Postgres implementations

`postgres_metrics_store.py`: `register` ON CONFLICT idempotent. `increment` / `set` / `observe` write evt_monitoring_metric_points; histogram.observe buckets the value against `histogram_buckets` from the registry. `postgres_spans_store.py`: `insert_batch` executemany; `query_by_trace` returns all spans sorted by start_time then parent relationship.
`postgres_resources_store.py`: `upsert` computes sha256 hash of canonical JSON; ON CONFLICT (org_id, resource_hash) DO UPDATE RETURNING id.
