# Plan 13-03 — OTLP Receiver + Auto-Instrumentation

**Phase:** 13 — Monitoring
**Goal:** Accept OpenTelemetry logs + traces over OTLP/HTTP (binary protobuf AND JSON) at `POST /v1/monitoring/otlp/v1/logs` and `POST /v1/monitoring/otlp/v1/traces`, publish each ResourceLogs/ResourceSpans to NATS JetStream. Auto-instrument FastAPI and asyncpg so every inbound request + DB call emits spans + latency metrics without developer effort. Wire a structlog bridge so backend logs automatically flow into the receiver.

---

<objective>

## Goal
Make logs and traces enter the system through standard OTel tooling (OTel Collector, SDKs, direct HTTP) and make the backend itself the first producer via auto-instrumentation.

## Purpose
The receiver is the public contract — every OTel client in the world can push here. Auto-instrumentation means day-one dogfooding: the monitoring system observes itself, proving correctness.

## Output
- 2 OTLP endpoints (logs, traces), each accepts `Content-Type: application/x-protobuf` AND `application/json`
- Receiver decodes OTLP → publishes to JetStream (no Postgres write here — that's 13-04's consumer)
- OTLP partial-success response envelope per spec
- FastAPI middleware: every request emits a server-kind span + request latency histogram + request count counter
- asyncpg hook: every query emits a client-kind span + query latency histogram
- structlog → monitoring bridge: backend log calls automatically publish OTel logs through the receiver (local bypass of HTTP — direct to JetStream publish)
- ≥15 pytest tests green

</objective>

## AC-1: OTLP Logs receiver
```gherkin
Given a valid OTLP/HTTP logs protobuf payload
When POST /v1/monitoring/otlp/v1/logs with Content-Type: application/x-protobuf
Then response is 200 with OTLP ExportLogsServiceResponse (empty partial_success on happy path)
And N NATS messages are published to subject monitoring.logs.otel.{service_name} — one per ResourceLogs × ScopeLogs × LogRecord batch
And application/json body also works with same semantics
And malformed body returns 400 with partial_success.rejected_log_records = count
And unauthenticated requests return 401 unless bearer token matches TENNETCTL_MONITORING_OTLP_TOKEN env (vault-stored per project config rule — env var references vault key)
```

## AC-2: OTLP Traces receiver
```gherkin
Given a valid OTLP/HTTP traces payload
When POST /v1/monitoring/otlp/v1/traces
Then NATS publishes to monitoring.traces.otel.{service_name} — one message per ResourceSpans batch
And 200 + ExportTraceServiceResponse; bad payload → 400 + rejected_spans count
```

## AC-3: FastAPI auto-instrumentation
```gherkin
Given instrumentation is enabled via config.MONITORING_AUTO_INSTRUMENT=true
When any HTTP request hits the backend
Then middleware:
  - Creates a server-kind span with attributes (http.method, http.route, http.status_code, http.target)
  - Records histogram monitoring.http.server.duration_seconds (buckets: [0.005, 0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1, 2.5, 5, 10])
  - Increments counter monitoring.http.server.requests_total {method, route, status_class}
  - Propagates trace context from traceparent header or creates new trace_id
  - Span end publishes directly to JetStream monitoring.traces.otel.tennetctl-backend
And the metrics are auto-registered on first request (via SDK lazy register)
And endpoints under /v1/monitoring/otlp/** are excluded from instrumentation (prevents infinite loop)
```

## AC-4: asyncpg auto-instrumentation
```gherkin
Given instrumentation is enabled
When any asyncpg connection executes a query
Then a client-kind span is created with attributes (db.system=postgresql, db.statement=<redacted>, db.name)
And histogram monitoring.db.client.duration_seconds records query time
And db.statement is truncated to 256 chars and stripped of literal values (regex-replace with "?") to avoid leaking PII into traces
And the instrumentation uses asyncpg's built-in `conn.add_query_logger` or equivalent hook
```

## AC-5: structlog bridge
```gherkin
Given backend code does `log.info("signin", user_id=..., org_id=...)`
When the monitoring bridge is installed
Then the log entry is also published to JetStream monitoring.logs.otel.tennetctl-backend
  - severity_id resolved from log level (info→9, warning→13, error→17)
  - body = log message
  - attributes = structlog event dict (minus noise keys)
  - trace_id / span_id injected from contextvars if active span exists
And the bridge does NOT round-trip via HTTP (direct JetStream publish)
And the bridge failure (NATS down) must NOT crash the log call (silent drop + inc of monitoring.bridge.drops_total)
```

## AC-6: No infinite loops
```gherkin
Given instrumentation + structlog bridge are active
When a log is published from inside instrumentation code
Then no recursion occurs — bridge uses a contextvar flag "_in_monitoring_bridge" to short-circuit
```

## Task 1 — OTLP decoder
`sub_features/01_logs/otlp_decoder.py` + `sub_features/03_traces/otlp_decoder.py`:
- Use `opentelemetry.proto.collector.logs.v1.logs_service_pb2` + `trace_service_pb2` for protobuf
- JSON decode via protobuf `json_format.Parse`
- Returns a list of (subject, payload_bytes) tuples for JetStream publishing — keep payloads as the original OTel protobuf so downstream consumer in 13-04 decodes from source (no double-decode)

## Task 2 — Receiver routes
`sub_features/01_logs/routes.py`:
```python
@router.post("/otlp/v1/logs", status_code=200)
async def otlp_logs(request: Request) -> Response:
    # auth check (bearer token from vault)
    body = await request.body()
    ctype = request.headers.get("content-type", "application/x-protobuf")
    batches, rejected = decode_logs(body, ctype)
    for subj, payload in batches:
        await js.publish(subj, payload)
    return build_partial_success(len(batches), rejected)
```
Mirror for traces. Routes mount under `/v1/monitoring` (NOT `/v1/monitoring/otlp/v1/logs` — actual OTLP spec path is `/v1/logs` but we prefix for namespace safety; document convention).

## Task 3 — Auth
OTLP receiver auth via bearer token. Token value lives in vault under key `monitoring.otlp.ingest_token`. Middleware resolves at request time; cache 60s.

## Task 4 — FastAPI instrumentation
`instrumentation/fastapi.py`:
- `install(app, config)` — registers middleware
- Middleware wraps each request: start span (extract traceparent), record start timer
- On response: set span status from HTTP status, record histogram, increment counter, publish span to JetStream directly (not via HTTP — local bypass)
- Skip instrumentation for paths starting with `/v1/monitoring/otlp/` + `/health` + `/docs`
- Uses a `Span` dataclass + direct JetStream publish with protobuf encoding

## Task 5 — asyncpg instrumentation
`instrumentation/asyncpg.py`:
- `install(pool)` — patches pool setup to attach `server_settings` and a `connect` hook that installs `_async_query_execute` wrapper
- Redacts literal values in SQL via regex replacing `'...'` and numeric literals with `?`
- Publishes client-kind span per query

## Task 6 — structlog bridge
`instrumentation/structlog_bridge.py`:
- `install(logger)` — adds a structlog processor that after formatting calls the bridge
- Bridge encodes to OTLP LogRecord protobuf; publishes to JetStream
- Uses `contextvars.ContextVar("_in_monitoring_bridge", default=False)` to prevent recursion
- Silent drop + counter inc on NATS failure

## Task 7 — Lifespan wiring
