<objective>
## Goal
Ship a vault-backed, per-key auth policy configuration layer with a global-plus-per-org resolver, in-process SWR cache, and immediate invalidation on writes — so every subsequent plan in Phase 20 reads its thresholds, TTLs, and limits through `AuthPolicy` instead of hardcoding them.

## Purpose
Pre-OSS audit found ~20 IAM thresholds hardcoded across `11_magic_link/`, `12_otp/`, `09_sessions/`, `08_credentials/`, and `14_password_reset/`. Operators need to tune these without a code change or restart, and enterprises need per-org overrides. This plan is the foundation — no auth flow behavior changes yet, only the config pipe that plans 20-03…20-06 will read from.

## Output
- `AuthPolicy` service on `app.state.auth_policy` that resolves any `iam.policy.{key}` with per-org override fallback to global, cached 60s per `(org_id, key)` tuple, and invalidated inline on vault config writes that touch `iam.policy.*`.
- Domain-grouped getters: `PasswordPolicy`, `SessionPolicy`, `LockoutPolicy`, `MagicLinkPolicy`, `OtpPolicy`, `PasswordResetPolicy` (frozen dataclasses) — subsequent plans call `await app.state.auth_policy.password(org_id)` etc.
- Idempotent bootstrap that seeds 20 safe-default global policy entries on boot (reads env-free; inserts via `vault.configs` service to preserve audit trail).
- Pytest coverage: resolver precedence, cache TTL, cache invalidation on vault write, bootstrap idempotency, type-safe getter dataclasses.
</objective>

```gherkin

And `AuthPolicy.resolve(org_id=ORG_B, key="password.min_length")` returns 12 (global fallback)

```

```gherkin

Then the DB is hit exactly once for the (org_id, key) pair

```

## AC-3: Immediate cache invalidation on vault config writes that touch iam.policy.*

```gherkin

Then `AuthPolicy.invalidate(org_id, key)` is called synchronously inside the vault.configs service tx
And the next `resolve()` call for that (org_id, key) hits the DB

```

## SCOPE LIMITS
- No admin UI — `/iam/security/policy` page ships in plan 20-02.
- No consumer rewiring — `11_magic_link/`, `12_otp/`, `09_sessions/`, `08_credentials/`, `14_password_reset/` keep their hardcoded values in this plan. Plans 20-03…20-06 delete those constants.
- No passkey policy keys, no rate-limit policy keys, no HIBP config, no 2FA-required enforcement keys — all explicitly deferred to v0.1.8 per ROADMAP.
- No workspace-level override — only global + per-org resolution. Workspace layer is a v0.2 concern.
- No Valkey cache — the in-process SWR cache is sufficient for this size (20 keys × N orgs); Valkey fallback is available in monitoring but introduces a new dependency for a non-hot-path lookup.
