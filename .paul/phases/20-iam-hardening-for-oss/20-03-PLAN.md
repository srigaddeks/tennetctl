<objective>
## Goal
Wire account lockout into the credentials signin path using policy-driven thresholds from 20-01. Every failed signin increments a sliding-window counter; on threshold breach, the user (and/or source_ip) is locked until the lockout window expires. Emit `iam.lockout.triggered` and `iam.lockout.cleared` audit events.

## Purpose
The pre-OSS audit found no account lockout on credentials signin (only email-OTP has it today). This is the single largest brute-force surface. All thresholds come from `AuthPolicy.lockout(org_id)` — zero hardcoded values.
</objective>

## AC-2: Failed signin increments counter + emits audit
```gherkin

```

## AC-3: Lockout triggers on threshold breach
```gherkin
Given 4 failed attempts for email X within the last 900 seconds
When attempt 5 fails
Then the service sets `lockout.until = now() + 900s` as a dtl_attr on the user record
And emits `iam.lockout.triggered` audit event with metadata {email, attempts, until}
And subsequent signin attempts for that email within 900s return 423 Locked with ok=false
```

## AC-4: Expired lockout auto-clears on next signin attempt
```gherkin
Given a user is locked (lockout.until < now())
When they successfully authenticate
Then the dtl_attr is cleared in the same tx as session creation
And `iam.lockout.cleared` audit event is emitted
```

## AC-5: Per-org thresholds respected
```gherkin
Given ORG_A has lockout.threshold_failed_attempts=3 (override) and ORG_B uses global=5
When a user in ORG_A fails 3 times
Then they are locked
And a user in ORG_B at 3 failures is NOT locked
```
