<objective>
## Goal
Close the remaining pre-OSS items: (1) TOTP backup codes, (2) migrate email OTP delivery to the existing Notify transactional pipe (remove any inline SMTP), (3) API key rotation endpoint with `last_used_at` tracking in Bearer middleware, (4) IAM metrics emissions.

## Purpose
Four independent follow-ons consolidated because each is small and none blocks the others. Backup codes give users account recovery when phones are lost. Notify migration removes a duplicate email path. API key rotation supports secret hygiene. Metrics give monitoring dashboards real signal beyond audit events.
</objective>

## AC-1: TOTP backup codes — schema + generate + consume
```gherkin
Given migration applies
Then "03_iam"."fct_totp_backup_codes" exists with (id UUID v7, user_id UUID, code_hash TEXT argon2id, consumed_at TIMESTAMP NULL, created_at)
When a user enrolls TOTP
Then service generates 10 one-time codes, returns them ONCE to the enroll response (plaintext shown to user), stores argon2id hashes
When the same code is submitted to /auth/otp/verify as a totp fallback
Then it's marked consumed_at and cannot be reused; single-use enforced by partial unique index on (user_id) where consumed_at IS NULL is NOT the pattern — uniqueness is per code_hash
When all 10 codes are consumed
Then the service surfaces a "regenerate backup codes" path (new endpoint) that issues 10 new codes and deletes old ones
```

## AC-2: Email OTP routes through existing Notify template
```gherkin
Given a new notify template seeded with key `iam.otp.email` in the transactional template group
When OTP service sends an email OTP
Then it calls `POST /v1/notify/send` via the existing transactional node / in-process client
And no inline aiosmtplib / SMTP call remains in 12_otp/service.py
And the email renders with variables {otp_code, ttl_minutes, user_display_name}
```

## AC-3: API key rotation + last_used_at
```gherkin
Given migration adds fct_api_keys.last_used_at TIMESTAMP NULL
When a request successfully auths via Bearer token
Then middleware updates last_used_at = CURRENT_TIMESTAMP (fire-and-forget)
When POST /v1/api-keys/{id}/rotate is called
Then the old row is marked revoked in the same tx as a new row created with the same scopes
And the new token is returned ONCE in the response body
And audit event `iam.api_keys.rotated` is emitted
```

## AC-4: IAM metrics emitted

```gherkin
  iam_otp_verify_total{kind,outcome},
  iam_password_reset_total{outcome}
And a periodic task (every 30s) emits iam_active_sessions gauge via monitoring.metrics.gauge (or increment+reset pattern if gauge node isn't available)
```
