<objective>
## Goal
Stand up the audit event analytics foundation: a typed taxonomy (`dim_audit_event_keys` + `dim_audit_categories`), a read-path `v_audit_events` view, a PostHog-class query API (list + detail + stats + registered-keys) with cursor pagination and rich filtering, and an `audit.events.query` control node. All covered by pytest.

## Purpose
Phase 3 Plan 03 gave us the write path (`audit.events.emit`) and the evt_audit table. The store is a black box without a read surface. This plan opens the box: every downstream feature (Audit Explorer UI, Notify subscriptions, exports, webhooks) needs a single well-defined query API. Typed taxonomy replaces free-text CHECK-constraint strings so UIs can render labels/descriptions and features can self-register the event keys they emit.

## Output
- 1 migration: dim_audit_event_keys + dim_audit_categories + FK migration on evt_audit + `v_audit_events` view
- 1 seed: 04audit_categories.yaml (4 rows: system/user/integration/setup)
- 4 audit sub-feature source files (schemas/repository/service/routes) implementing list + detail + stats + keys endpoints
- 1 node handler: `audit.events.query` (control, read-only)
- Manifest + router wiring
- TS types in `frontend/src/types/api.ts`
- Pytest suite covering all filter combinations, pagination, stats, authz scope
</objective>

## Architectural Decisions to Respect
- Pure EAV rule (no business columns on fct_*) — taxonomy lives in dim_* tables, not on evt_audit
- Views use `MAX(key_text) FILTER (WHERE ad.code = ...)` EAV pivot — not needed here since evt_audit is native columns, but v_audit_events still resolves dim codes
- Reads → view; writes → raw tables
- Node keys require 3+ segments: `audit.events.query` (not `audit.query`)
- Control nodes may perform read-only DB ops (Phase 4 Plan 01 widening)
- HTTP reads audit themselves, node reads do not (vault precedent, Plan 07-01)
- Cursor pagination on UUID v7 ordering — `id < cursor AND created_at <= created_at_of_cursor` pattern
</context>
