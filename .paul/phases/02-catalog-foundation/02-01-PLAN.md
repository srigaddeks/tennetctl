## Goal
Lay the Node Catalog Protocol (NCP v1) foundation: (1) publish the protocol document, (2) record ADR-027, (3) create the `"01_catalog"` Postgres schema with all tables from NCP §5 and seed the dim tables that the manifest loader and runner will depend on.

## Purpose
Phase 2 Plans 02-02 (manifest loader + validator) and 02-03 (node runner) need the catalog tables to exist before they can upsert anything. Without this plan, no Python code in Phase 2 can run. Phase 3+ feature verticals register themselves into this catalog via their `feature.manifest.yaml`. Getting the schema and seeds right here avoids costly ripple migrations later.

## Database Conventions (from CLAUDE.md + rules/database.md)
Table types:
- `dim_*` (01–09): lookup enums, SMALLINT PK, seeded via YAML, never mutated. Mandatory cols: id, code, label, description, deprecated_at.
- `fct_*` (10–19): entity identity. For catalog, PK is SMALLINT (identity generated), not UUID — catalog entities are system-level, not user data. Mandatory cols: id, is_active, is_test (both default matching catalog semantics), deleted_at, created_by, updated_by, created_at, updated_at.
- `dtl_*` (20–29): EAV attributes. UUID v7 PK for `dtl_attrs`, SMALLINT for `dtl_attr_defs`.

Critical:
- No business columns on fct_* — all variable attrs go through dtl_attrs via EAV
- NEVER Postgres ENUMs — use dim_* tables
- TIMESTAMP not TIMESTAMPTZ — app always passes UTC
- CURRENT_TIMESTAMP not now()
- deleted_at TIMESTAMP — not is_deleted BOOLEAN
- Partial unique index for soft-delete-aware uniqueness
- All constraints explicitly named: pk_, fk_, uq_, chk_, idx_
- COMMENT ON every table and column
- updated_at set by app in every UPDATE — no triggers

    **Migration 003 — fct tables for features, sub-features, nodes.**

    Catalog fct PKs use SMALLINT GENERATED ALWAYS AS IDENTITY (not UUID v7) because:
    - Catalog entities are system-level, not user data
    - SMALLINT saves space in the large lnk tables that will reference them later
    - Deterministic IDs make boot upserts simpler

    Mandatory cols on catalog fct tables (no audit columns — catalog is system-managed):
    - `id SMALLINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY`
    - `key TEXT NOT NULL UNIQUE` (the stable NCP key)
    - `deprecated_at TIMESTAMP` (nullable)
    - `tombstoned_at TIMESTAMP` (nullable, ≥180 days after deprecated_at per NCP §12)
    - `created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP`
    - `updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP`

    Tables to create:

    - `"01_catalog"."10_fct_features"`
      Adds:
      - `number SMALLINT NOT NULL` (permanent feature slot 01–99)
      - `module_id SMALLINT NOT NULL` FK→dim_modules
      Constraints:
      - uq_fct_features_number(number)
      - uq_fct_features_module_id(module_id) — one feature per module in v1 (feature.key == module name)

    - `"01_catalog"."11_fct_sub_features"`
      Adds:
      - `feature_id SMALLINT NOT NULL` FK→fct_features
      - `number SMALLINT NOT NULL` (sub-feature slot 01–99 within feature)
      Constraints:
      - uq_fct_sub_features_feature_number(feature_id, number)

    - `"01_catalog"."12_fct_nodes"`
      Adds:
      - `sub_feature_id SMALLINT NOT NULL` FK→fct_sub_features
      - `kind_id SMALLINT NOT NULL` FK→dim_node_kinds
      - `handler_path TEXT NOT NULL` (e.g. `sub_features.01_orgs.nodes.iam_orgs_create.CreateOrg`)
      - `version SMALLINT NOT NULL DEFAULT 1`
      - `emits_audit BOOLEAN NOT NULL DEFAULT false`
      - `timeout_ms INTEGER NOT NULL DEFAULT 5000 CHECK (timeout_ms BETWEEN 100 AND 600000)`
      - `retries SMALLINT NOT NULL DEFAULT 0 CHECK (retries BETWEEN 0 AND 3)`
      - `tx_mode_id SMALLINT NOT NULL` FK→dim_tx_modes
      Constraints:
      - chk_effect_must_emit_audit: CHECK (kind_id <> 2 OR emits_audit = true) — effect nodes must emit audit
      - idx_fct_nodes_sub_feature(sub_feature_id) — lookup by parent

    - COMMENT ON every table and every column
    - DOWN drops all three in reverse dependency order

    Avoid:
    - UUID v7 PKs on catalog fct tables (use SMALLINT identity — see rationale above)
    - Adding business/user columns (labels, descriptions) — those go in dtl_attrs via EAV
    - Foreign keys without ON DELETE RESTRICT (default; explicit prevents silent cascade)
  </action>
