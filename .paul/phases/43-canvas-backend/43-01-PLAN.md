# Plan 43-01 — Canvas Renderer Backend (trace assembly, port resolution)

**Phase:** 43 — Canvas + Visual Flow Viewer (v0.4.0)
**Goal:** Serve a single render-payload endpoint the React Flow client (Plan 44-01) consumes: `(flow_version_id) -> {nodes, edges, ports, layout, trace?}`. Port types are resolved live from the catalog registry. Execution traces are joined from the existing flow run event stream and overlaid on node/edge IDs without modifying the run pipeline.

---

<objective>

## Goal
Add a `canvas` sub-feature to the catalog feature that owns one purpose: produce the JSON the React Flow renderer needs in one round-trip. The endpoint reads the locked `fct_catalog_flow_versions` + dtl rows from Plan 42-01, resolves every node's input/output ports from the live in-process registry (the same `client.catalog` surface external apps query — no duplication, no copy), computes a deterministic auto-layout via topological levels + lane packing, and — when a `?trace_id=` is passed — joins the matching `evt_catalog_flow_run_events` rows to produce per-node status (pending/running/success/failure/skipped) plus per-edge `traversed: bool`. The endpoint is read-only. No write paths.

## Purpose
The canvas exists to let developers **trace** what actually happened during a flow run, not just see the static graph. Bolting trace assembly into the run-event writer would couple read-time presentation to write-time hot path. A separate read-side assembler keeps the run pipeline untouched, lets the canvas evolve independently, and gives Plan 44-01 a single endpoint contract instead of three (graph + ports + trace).

## Output
- `dim_catalog_trace_node_status` (SMALLINT PK seeded: pending=1, running=2, success=3, failure=4, skipped=5, timed_out=6)
- View `v_catalog_flow_run_node_status` — projection over `evt_catalog_flow_run_events` returning the **latest** status per `(flow_run_id, node_instance_id)` via `DISTINCT ON` ordered by `occurred_at DESC` so the assembler reads one row per node
- View `v_catalog_flow_run_edge_traversal` — projection that marks an edge `traversed=true` when its `from_node_id` reached `success` and the corresponding `to_node_id` has any event for the same run
- `port_index.py`: pure `build_port_index(node_keys: list[str], registry) -> dict[str, ResolvedPorts]` — single bulk registry walk, no per-node round trips, returns `{node_key: {inputs, outputs}}`
- `layout.py`: pure `compute_layout(nodes, edges) -> dict[node_id, {x, y, lane}]` — topological-level x assignment (level × 240px), greedy lane packing for y (lane × 120px), stable sort by `instance_label` within a level for determinism. Honors operator-set `position_x/position_y` when present (operator wins, layout fills nulls)
- `trace_assembler.py`: pure `assemble_trace(run_node_rows, run_edge_rows, nodes, edges) -> CanvasTrace` — returns `{node_status: {node_id: status}, edge_traversed: {edge_id: bool}, started_at, finished_at, total_duration_ms}` using already-loaded rows; no I/O
- Routes (1 + 1 trace list, both read-only):
  - `GET /v1/flows/{flow_id}/versions/{version_id}/canvas?trace_id=<optional>` — single payload; `trace_id` is optional. Returns 404 if version belongs to a different flow_id than the path, 200 with `trace=null` if `trace_id` omitted
  - `GET /v1/flows/{flow_id}/versions/{version_id}/runs?from&to&status&limit=50` — list recent runs for the run picker the canvas dropdown shows. NO new run-record table; reads the existing `fct_catalog_flow_runs`
- ≥16 pytest, no E2E in this plan

## AC-1: Single-call canvas payload contains everything the renderer needs
```gherkin
Given a published flow version with 5 nodes and 6 edges
When client GETs /v1/flows/{fid}/versions/{vid}/canvas
Then response.data has exactly the keys: nodes, edges, ports, layout, trace
And nodes is a list of 5 entries each with {id, instance_label, node_key, kind, config_json, position}
And edges is a list of 6 entries each with {id, from_node_id, from_port, to_node_id, to_port, kind}
And ports is a map node_key→{inputs:[{key,type}], outputs:[{key,type}]} resolved from the live registry
And layout is a map node_id→{x,y,lane} with x increasing by topological level
And trace is null because no trace_id was provided
And the entire payload is served in a single DB transaction (one acquire from the pool)
```

## AC-2: Topological auto-layout is deterministic and honors operator positions
```gherkin
Given a flow with nodes A→B, A→C, B→D, C→D where A has position={x:50,y:50} set by operator
When compute_layout runs
Then A.x=50, A.y=50 (operator wins)
And B and C are placed at level 1 (x=240) on lanes 0 and 1 (y=0, y=120)
And D is placed at level 2 (x=480, y=0)
And re-running compute_layout on the same input returns byte-identical coordinates
And nodes with the same level are sorted by instance_label so order is stable across calls
```

## AC-3: Trace overlay maps run events to per-node + per-edge state
```gherkin
Given a flow version with nodes auth→handler→audit and a flow_run with events:
  - auth: started, succeeded
  - handler: started, succeeded
  - audit: started (no terminal event yet)
When client GETs .../canvas?trace_id=<run_id>
Then trace.node_status = {auth:"success", handler:"success", audit:"running"}
And trace.edge_traversed = {auth→handler:true, handler→audit:true}
And trace.started_at = earliest event occurred_at
And trace.finished_at = null (audit still running)

Given a flow_run where auth succeeded but handler failed
Then trace.node_status = {auth:"success", handler:"failure", audit:"skipped"}
And trace.edge_traversed = {auth→handler:true, handler→audit:false}
And the failure edge sibling (handler→error_handler if defined) is marked traversed=true
```

## AC-4: Port resolution is live from the registry, never persisted
```gherkin
Given a flow version that references node_key "string.upper" with input port "value":string
And the operator deploys a code change widening "value" to type=any
When client GETs the canvas payload
Then ports["string.upper"].inputs[0].type = "any" — reflects the new code, no migration needed
And no row in any dtl_* table has been touched
And the assembler issued exactly one bulk registry call (port_index.build_port_index), not one call per node
And a node_key absent from the registry returns ports[node_key] = {inputs:[], outputs:[], unresolved:true} so the canvas can render a red "missing node" badge
```

## Task 1 — Migration + seed + read views
`20260422_081_catalog-canvas-trace.sql`:
- `dim_catalog_trace_node_status` SMALLINT PK seeded
- `v_catalog_flow_run_node_status`: `SELECT DISTINCT ON (flow_run_id, node_instance_id) ... ORDER BY flow_run_id, node_instance_id, occurred_at DESC`
- `v_catalog_flow_run_edge_traversal`: lateral join from `dtl_catalog_flow_edges` to the latest event of `from_node_id`; `traversed = (from_status IN ('success','failure_handled') AND to_node has any event)`

Seed:
- `01catalog_05_dim_trace_node_status.yaml`

NOTE: this plan adds **views only** over the existing `evt_catalog_flow_run_events` table. The events table itself was created in a prior phase (flow execution); we do not modify it.

## Task 2 — Pure compute modules
- `port_index.py`:
  - `build_port_index(node_keys: list[str], registry) -> dict[str, ResolvedPorts]` — dedup keys, single registry traversal, returns map; missing key returns sentinel `unresolved=true`
  - reuses `port_resolver.resolve_ports` from Plan 42-01 internally — no logic duplication
- `layout.py`:
  - `topological_levels(nodes, edges) -> dict[node_id, int]` — Kahn-derived level for each node (longest path from any source)
  - `compute_layout(nodes, edges) -> dict[node_id, Position]` — level × 240 for x, greedy lane packing within level for y, stable sort by `instance_label`, operator-set positions win
- `trace_assembler.py`:
  - `assemble_trace(node_status_rows, edge_traversal_rows, version_nodes, version_edges) -> CanvasTrace` — pure join; computes started_at = min(event time), finished_at = max if every node terminal, total_duration_ms = (finished_at - started_at).total_milliseconds()
  - Skipped logic: any node downstream of a failed node with no event of its own → `skipped`

All three modules conn-free. The route layer does the I/O.

## Task 3 — Sub-feature scaffold + repository + service
`sub_features/05_canvas/` exactly 5 files + `port_index.py` + `layout.py` + `trace_assembler.py`. Repository:
- `load_version_dag(conn, flow_id, version_id) -> FlowVersionDetail | None` — reuses query from `04_flows.repository.get_version` via shared helper, no copy-paste
- `load_run_node_status(conn, flow_run_id) -> list[RunNodeStatus]` — reads `v_catalog_flow_run_node_status`
- `load_run_edge_traversal(conn, flow_run_id) -> list[RunEdgeTraversal]` — reads `v_catalog_flow_run_edge_traversal`
- `list_runs(conn, version_id, filters) -> list[FlowRunSummary]` — reads existing `fct_catalog_flow_runs` filtered by version_id

Service `assemble_canvas(conn, flow_id, version_id, trace_id=None) -> CanvasPayload`:
- Single conn, single tx (set transaction read-only)
- Loads dag, resolves ports via `port_index.build_port_index(distinct(node_keys))`, computes layout, optionally loads trace rows and assembles trace
- Returns ready-to-serialize `CanvasPayload`

importlib for `backend.01_core.database`, `backend.01_core.id`, `backend.01_core.response`. Conn passed in, never `pool.acquire()` outside the route.

## Task 4 — Routes (read-only, 2 endpoints)
`routes.py`:
- `GET /v1/flows/{flow_id}/versions/{version_id}/canvas` query: `trace_id?: UUID` — calls `service.assemble_canvas`, returns envelope `{ok:true, data: CanvasPayload}`. Returns 404 with code `FLOW_VERSION_NOT_FOUND` if mismatch; 200 with `trace=null` when no trace_id
- `GET /v1/flows/{flow_id}/versions/{version_id}/runs` query: `from?, to?, status?, limit=50` — paginated list of run summaries for the trace picker

Cache: response is safely `Cache-Control: private, max-age=2` when no trace_id (DAG is immutable for published versions). Trace responses are uncached (state evolves while running).
