# Plan 44-01 ‚Äî Canvas Renderer Frontend (React Flow UI, trace overlay, search)

**Phase:** 44 ‚Äî Canvas + Visual Flow Viewer (v0.4.0)
**Goal:** Ship a read-only React Flow canvas at `/flows/[id]/versions/[versionId]` that renders the payload from Plan 43-01, overlays an execution trace selected from a run picker, supports node/port search with focus + highlight, and degrades gracefully when an upstream node_key has no live registry entry.

---

<objective>

## Goal
Add a `flows` route group to the Next.js app router. The canvas page hydrates from `GET /v1/flows/{id}/versions/{versionId}/canvas?trace_id?`, transforms the payload into React Flow's `nodes` + `edges` shape, and renders custom node + edge components that show node kind (request/effect/control), input/output port handles colored by port type, and trace status (pending/running/success/failure/skipped/timed_out) when a trace is active. A run picker dropdown lists recent runs from `GET .../runs`. A search box matches against `instance_label`, `node_key`, and port keys; selecting a result pans to and outlines the node. A node click opens an inspector showing config_json + ports + last-run status.

## Purpose
The visual canvas is the user-facing payoff for the entire node-first architecture. Until developers can see the DAG and trace what happened in a run, the abstractions in ADR-016 / 017 / 018 stay invisible. Read-only is intentional ‚Äî editing is a separate later milestone; building the viewer first lets us validate the schema (Plan 42-01) and assembler (Plan 43-01) under real eyes before adding write affordances.

## Output
- `package.json` adds `reactflow` (XY Flow per ADR-023) ‚Äî pinned to current stable
- TanStack Query hooks (read-only):
  - `use-flows.ts` ‚Äî `useFlows(filters)`, `useFlow(id)`
  - `use-canvas.ts` ‚Äî `useCanvas(flowId, versionId, traceId?)` ‚Äî `staleTime: 60_000` when traceId absent (DAG immutable for published), `refetchInterval: 2000` when traceId present
  - `use-flow-runs.ts` ‚Äî `useFlowRuns(flowId, versionId, filters)` for the picker
- Pages:
  - `/flows` ‚Äî flow list (table: name, slug, status, version count, updated_at)
  - `/flows/[id]` ‚Äî flow detail with version sidebar; latest published version preselected
  - `/flows/[id]/versions/[versionId]` ‚Äî full-bleed canvas viewer
- Components (single-purpose each):
  - `flow-list.tsx` ‚Äî paginated table, search by slug
  - `canvas-viewer.tsx` ‚Äî React Flow `<ReactFlow>` host; mounts background, controls, minimap; passes nodes/edges through `canvas-transform.ts`
  - `canvas-node.tsx` ‚Äî custom node: kind badge (request/effect/control), instance_label, node_key sub-label, input handles on left, output handles on right, status ring around the node when trace active
  - `canvas-edge.tsx` ‚Äî custom edge with stroke color by `kind` (next=gray, success=green, failure=red, true_branch=blue, false_branch=purple), animated when `traversed=true && trace.finished_at == null`, dashed when `traversed=false && trace !== null`

  - `canvas-trace-picker.tsx` ‚Äî dropdown listing recent runs (status badge + started_at relative time); selecting one updates `?trace_id=` in URL
  - `canvas-trace-legend.tsx` ‚Äî legend strip: status colors, edge kinds, port type colors
  - `canvas-node-inspector.tsx` ‚Äî slide-over panel on node click; shows instance_label, node_key, kind, config_json (json viewer), resolved ports, current trace status if any
- Lib:
  - `canvas-transform.ts` ‚Äî pure `toReactFlow(payload: CanvasPayload, trace: CanvasTrace | null) -> {nodes, edges}` ‚Äî applies layout positions, attaches data props each custom component reads
  - `port-color.ts` ‚Äî pure `colorFor(portType)` returning a tailwind color class (any=slate, string=blue, number=amber, boolean=green, object=violet, array=rose, uuid=cyan, datetime=teal, binary=stone, error=red)
- Feature flag: `flows.canvas_v1` in `frontend/src/config/features.ts` defaulting on for `core` mode
- One Playwright MCP walkthrough script captured in this plan's verification (NOT a `.spec.ts` file, NOT robot)

<acceptance_criteria>

## AC-1: Canvas hydrates and renders all nodes + edges from one fetch
```gherkin
Given a published flow version with 5 nodes and 6 edges exists
When the user navigates to /flows/{id}/versions/{versionId}
Then exactly one network call to /v1/flows/{id}/versions/{versionId}/canvas is observed
And React Flow renders 5 node DOM elements with the correct instance_label and node_key text
And 6 edge paths render with stroke colors matching their edge.kind
And input handles appear on the left of each node, output handles on the right
And handle colors match port_color.colorFor(port.type)
And the minimap shows all 5 nodes; controls show fit-to-view, zoom in/out
```

## AC-2: Run picker triggers trace overlay with status colors
```gherkin
Given the canvas is open on a flow version with prior runs
When the user opens the trace picker
Then the dropdown lists up to 50 recent runs with status badge and relative started_at
When the user selects a run where node "auth"=success, "handler"=success, "audit"=running
Then the URL updates to include ?trace_id=<run_id>
And node "auth" has a green status ring, "handler" green, "audit" pulsing blue (running)
And the edge auth‚Üíhandler animates green; handler‚Üíaudit animates blue; downstream untraversed edges are dashed gray
And the trace legend shows visible status + edge-kind keys
And the canvas re-fetches every 2s while the run is non-terminal; stops once trace.finished_at is set
```

## AC-3: Search focuses and outlines a node by label, node_key, or port
```gherkin
Given the canvas has nodes [auth (iam.auth_required), handler (core.run_handler), audit (audit.emit)]
When the user types "audit" in the canvas search box
Then the dropdown shows two matches: instance_label="audit" and node_key="audit.emit"
When the user picks the instance_label match
Then React Flow centers on that node at zoom 1.2
And the node renders a 2-second outline highlight, then fades
When the user types "in:value" (port-prefix syntax)
Then the dropdown shows every node with an input port keyed "value"
```

## AC-4: Inspector + missing-node degradation
```gherkin
Given the canvas is open
When the user clicks a node
Then a slide-over inspector opens showing instance_label, node_key, kind badge, config_json viewer, ports table (key, direction, type), and (if trace active) the node's current status + duration
And clicking outside or pressing Escape closes the inspector

Given the canvas payload contains a node whose ports.{node_key}.unresolved == true
Then that node renders with a red border and a "missing in registry" badge
And handles still render but appear gray/dashed
And the inspector for that node shows a yellow warning banner explaining the node_key is not loaded in this build
And no JavaScript error is thrown
```

## Task 1 ‚Äî Types + hooks + dependency
- Add `reactflow` (XY Flow) to `frontend/package.json`. Pin the major version. No experimental subpackages.
- Extend `frontend/src/types/api.ts` (additive ‚Äî types `Flow`, `FlowVersion`, `CanvasPayload`, `CanvasNode`, `CanvasEdge`, `CanvasLayoutEntry`, `CanvasTrace`, `TraceNodeStatus`, `FlowRunSummary` were added by Plans 42-01 and 43-01; verify they exist and import cleanly here)
- Hooks under `frontend/src/features/catalog/hooks/`:
  - `use-flows.ts` ‚Äî `useFlows({status?, q?})`, `useFlow(flowId)`
  - `use-canvas.ts` ‚Äî `useCanvas(flowId, versionId, traceId?)`. When `traceId` undefined: `staleTime: 60_000`, `refetchInterval: false`. When `traceId` set: `staleTime: 0`, `refetchInterval: (data) => data?.trace?.finished_at ? false : 2000`
  - `use-flow-runs.ts` ‚Äî `useFlowRuns(flowId, versionId, {status?, limit?})`
- Every hook checks `data.ok` and throws `data.error.message` on failure (CLAUDE.md envelope rule)
- No `any`. Use `unknown` and narrow.

## Task 2 ‚Äî Pure transform + color libs
`canvas-transform.ts`:
- `toReactFlow(payload, trace) -> {nodes: Node[], edges: Edge[]}`:
  - For each `payload.nodes[i]` create RF node with `id`, `type:"tennetNode"`, `position: payload.layout[id]`, `data: {label, nodeKey, kind, configJson, ports: payload.ports[node_key], status: trace?.node_status[id], unresolved: payload.ports[node_key]?.unresolved ?? false}`
  - For each `payload.edges[i]` create RF edge with `id`, `type:"tennetEdge"`, `source`, `target`, `sourceHandle: from_port`, `targetHandle: to_port`, `data: {kind, traversed: trace?.edge_traversed[id], hasTrace: trace !== null}`
- Pure, no side effects, exhaustive unit-coverable

`port-color.ts`:
- `colorFor(portType: PortType): {bg: string, ring: string}` ‚Äî returns tailwind class strings; complete switch over the 10 port types

## Task 3 ‚Äî Pages + flow list
- `/flows/page.tsx`: server component shell + client `<FlowList/>` calling `useFlows`. Table cols: name, slug, status badge, version count, updated_at relative. Click ‚Üí `/flows/[id]`
- `/flows/[id]/page.tsx`: flow detail; left rail listing versions (latest published preselected), main pane links to canvas
- `/flows/[id]/versions/[versionId]/page.tsx`: full-bleed (`h-screen w-screen`) canvas viewer. Reads `trace_id` from query string; passes to `useCanvas`. Hosts `<CanvasViewer/>`, `<CanvasSearch/>` (top-right), `<CanvasTracePicker/>` (top-left), `<CanvasTraceLegend/>` (bottom-left), and `<CanvasNodeInspector/>` (right slide-over)

Use existing dashboard layout shell. Match the table styling used in `slo-list.tsx`.

## Task 4 ‚Äî Custom node + edge + interaction components
`canvas-node.tsx`:
- Renders a card: kind badge (color by request=blue, effect=green, control=amber), instance_label (lg font), node_key (mono, sm, slate-500)
- Input handles rendered on `Position.Left`, output handles on `Position.Right`, one per port; handle background uses `colorFor(port.type).bg`
- When `data.status` defined: outer ring colored by status (pending=slate, running=blue+pulse animation, success=emerald, failure=red, skipped=zinc, timed_out=orange)
- When `data.unresolved`: red border, "missing in registry" pill in top-right corner, gray handles
- onClick ‚Üí opens inspector via shared store (zustand, scoped to canvas page)

`canvas-edge.tsx`:
- Bezier path with stroke color per `data.kind`
- Animated dashes when `data.traversed && trace.finished_at === null`
- Solid color when `data.traversed && finished_at !== null`
- Dashed gray when `data.hasTrace && !data.traversed`
- Plain default when `!data.hasTrace`

`canvas-search.tsx`:
- Controlled input (top-right). Local state holds the canvas payload (lifted from page or read via store)
- Match modes:
  - default substring against instance_label and node_key
  - prefix `in:` matches input port keys; prefix `out:` matches output port keys
- Result list (max 8) shows match type icon + node label
- On select: calls React Flow `useReactFlow().setCenter(node.position.x, node.position.y, {zoom: 1.2, duration: 600})` and dispatches a 2s outline highlight

`canvas-trace-picker.tsx`:
- Dropdown driven by `useFlowRuns`. Each item: status pill, run id (short), `started_at` relative
- On select: `router.replace(\`?trace_id=\${runId}\`, {scroll:false})`
- "Clear trace" item resets to no overlay

`canvas-trace-legend.tsx`:
- Static legend strip: status row, edge kind row, port type row (only types present in current payload)

`canvas-node-inspector.tsx`:
- Slide-over panel (`fixed right-0 top-0 h-full w-96`); close on Esc, outside click, and explicit X
- Sections: Header (label + node_key + kind), Config (json-viewer over `data.configJson`), Ports (table key/direction/type colored), Trace (status + duration if active), Warnings (unresolved-node banner)
