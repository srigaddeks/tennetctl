<objective>
## Goal
Implement GDPR Articles 15 (access) + 17 (erasure): self-serve data export (async job → signed download URL emailed) and self-serve erasure request (30-day recovery window, then hard-erase purge job runs nightly).

## Purpose
GDPR compliance is a blocker for any EU deployment. OSS needs this primitive; self-hosters shouldn't bolt their own.
</objective>

## AC-2: Export flow
```gherkin
Given user U requests export
When POST /v1/account/data-export is called
Then a job row is inserted (kind=export, status=queued)
And a background worker within 60s assembles a JSON bundle (user row, orgs, workspaces, sessions, notifications, audit events actor=U)
And uploads to vault-configured storage (or local fs in v0.1.6 — configurable via policy `gdpr.storage_kind`)
And emails U a signed URL (Notify template iam.gdpr.export) valid 72h
And audit iam.gdpr.export_completed emitted
```

## AC-3: Erasure flow with recovery window
```gherkin
Given user U requests deletion
When POST /v1/account/delete-me is called with password confirm + TOTP if enrolled
Then a job row is inserted (kind=erase, status=pending, hard_erase_at=now()+30d)
And user is immediately soft-deleted per 21-04 flow (sessions revoked, pseudonymized)
And audit iam.gdpr.erase_requested emitted
When U signs back in within 30 days (via recovery email link)
Then the job is cancelled and user is un-pseudonymized (restored from audit metadata hash-reverse impossible — so recovery restores display_name and marks as recovered; email cannot be restored, user must update)
When 30 days pass without recovery
Then nightly purge job removes PII from evt_audit metadata where actor=U, hard-nulls related records, marks job status=completed
```

```gherkin
Then modal requires password + TOTP (if enrolled) + type "DELETE" to confirm
When export clicked
Then banner shows "Preparing — emailed when ready"
```
