<objective>
## Goal
Add signup email verification as a policy-gated flow: `signup.require_email_verification` (vault config from 20-01) defaults to `true`; unverified users can sign in but receive a restricted session + top-bar banner until they click the verification link in the email.

## Purpose
Current signup accepts any email with zero proof of ownership. For OSS credibility + preventing spam/typo registrations, verification must exist. Reuses Notify templates — no new mailer.

## Output
- New sub-feature `16_email_verification/` (5 files).
- `fct_email_verifications` table + new audit event keys.
- Notify template `iam.email.verify` seeded.
- `POST /v1/auth/verify-email/send` + `POST /v1/auth/verify-email/consume`.
- Signup hooks in: on create → schedule verification email; token consume → mark user verified.
- Frontend `/auth/verify-email?token=...` consumes token + redirects home.
- Robot E2E round-trip.
</objective>


## AC-2: Policy gate
```gherkin
Given AuthPolicy.resolve(org_id, "signup.require_email_verification") returns true
When a new user signs up
Then the auth service schedules a verification email via notify.send.transactional with template key iam.email.verify
And the user's dtl_attr `email_verified_at` is NULL
When the resolve returns false
Then no email is sent and email_verified_at is set to CURRENT_TIMESTAMP on signup
```

## AC-3: Consume flow
```gherkin
Given a valid unexpired verification token
When POST /v1/auth/verify-email/consume is called with the token
Then the matching row is marked consumed_at
And the user's email_verified_at dtl_attr is set to CURRENT_TIMESTAMP
And audit iam.email.verified is emitted
When the token is expired or already consumed
Then 400 returns with ok=false and iam.email.verification_failed audit emits
```

## AC-4: Unverified banner
```gherkin
Given user signed in with email_verified_at NULL
When GET /v1/users/me is called
Then response includes email_verified: false
And the topbar renders a banner with "Verify your email" + resend link
```
