<objective>
## Goal
SAML 2.0 SP implementation. Per-org IdP config (entity_id, metadata URL or XML, X.509 cert, attribute-to-user mapping). SP endpoints `/v1/auth/saml/{org_slug}/metadata`, `.../initiate`, `.../acs`. JIT user create/update on successful SAMLResponse. Admin UI for IdP configuration.

## Purpose
SAML is the single biggest enterprise wedge — any org with Okta/Azure AD/OneLogin demands it.

## Output
- Sub-feature `20_saml_sso/` (5 files + config handler).
- `python3-saml` dependency in pyproject.toml.
- Per-org SAML config in `fct_saml_configs` + dtl attrs for attribute mapping.
- SP private key + cert in vault (per-org, rotate-able).
- JIT provisioning creates users with `email_verified_at=CURRENT_TIMESTAMP` (IdP-proved).
</objective>

## AC-2: SP metadata endpoint
```gherkin
When GET /v1/auth/saml/{org_slug}/metadata
Then returns valid SAML metadata XML with SP entity ID, ACS URL, signing cert
```

## AC-3: Login flow round-trip
```gherkin
Given an IdP configured + test IdP (SimpleSAMLphp in docker)
When user visits /v1/auth/saml/acme/initiate
Then redirected to IdP SSO URL with signed AuthnRequest
When IdP POSTs valid SAMLResponse to /v1/auth/saml/acme/acs
Then response is validated (signature, audience, NotOnOrAfter)
And if user exists: updated attrs + session issued
And if not: JIT-created with email from mapped attr + auto-joined to org + default group
And audit iam.saml.login_succeeded emitted with {idp_entity_id, user_id, jit_created}
```

## AC-4: Failure modes
```gherkin
Given invalid signature / expired response / audience mismatch
When POST /.../acs
Then 403 with code=SAML_VALIDATION_FAILED
And audit iam.saml.login_failed with reason
```

## AC-5: Admin UI

```gherkin
And "Download SP metadata" + "Rotate SP certs" buttons
```
