<objective>
## Goal

Close the three remaining auth-hardening gaps on the OSS path: (1) session-fixation defense via session rotation on privilege-escalating events; (2) IP/fingerprint-based rate limiting on unauthenticated auth endpoints (distinct from per-user lockout); (3) single-use enforcement and tightened TTL on password-reset and magic-link tokens.

## Purpose

Per-user lockout (Plan 20-03) already defends against password brute-force targeting one account. What's still exposed:

- **Session fixation** — an attacker who injects a session cookie pre-login keeps that same session_id after the victim logs in, because nothing rotates the ID on auth success. Standard OWASP ASVS V3.2.1.
- **IP-level abuse** — lockout doesn't stop distributed enumeration (one attempt per account across many accounts) or stuffing via magic-link request spam. Need a global per-IP floor.
- **Token replay** — if a reset/magic-link token is captured from logs, email, or browser history, nothing prevents a second use. OWASP ASVS V2.5.1 requires single-use.

This closes Phase 38 auth hardening before Phase 39 NCP maturity.

## Output

- Session rotation: `rotate_session(conn, old_id) -> new_id` in 09_sessions, called from login success + password change + MFA enable paths
- Rate limiter: `auth_rate_limit` request-kind node keyed on `(ip, endpoint, window)` using existing Valkey client with Postgres fallback; applied as FastAPI dependency on `/v1/auth/login`, `/v1/auth/magic-link/request`, `/v1/auth/password-reset/request`
- Single-use tokens: `consumed_at TIMESTAMP` column on the two token tables; `consume_token` repo fn sets it atomically; TTL check tightened to 15 min (reset) / 10 min (magic-link)
- Audit events on each hardening boundary: `iam.session.rotated`, `iam.auth.rate_limited`, `iam.token.replay_blocked`
- Integration tests covering the four failure modes (replay, fixation, IP-burst, stale-token)
</objective>

<acceptance_criteria>

## AC-1: Session rotation on privilege escalation

```gherkin
Given an unauthenticated session_id S1 is planted in the browser (attacker-controlled)
When the victim successfully logs in using that browser
Then the response Set-Cookie carries a new session_id S2 (S2 != S1)
And the old session_id S1 is revoked in fct_sessions (deleted_at set)
And an audit event iam.session.rotated is emitted with reason='login'

And: when an authenticated user changes their password, session rotation fires with reason='password_change'
And: when an authenticated user completes MFA enrollment, session rotation fires with reason='mfa_enrolled'
And: the new session inherits user_id + org_id + workspace_id from the old one (no re-login required mid-flow)
```

## AC-2: IP-based rate limit on unauthenticated auth endpoints

```gherkin
Given the rate limit policy is 10 requests per IP per 60s on POST /v1/auth/login
When a single IP sends 11 POSTs within 60s (regardless of which email is used)
Then request 11 returns 429 with {ok:false, error:{code:"RATE_LIMITED", retry_after:N}}
And an audit event iam.auth.rate_limited is emitted with scope=(ip, endpoint, window)

And: same behavior on POST /v1/auth/magic-link/request and POST /v1/auth/password-reset/request
And: authenticated endpoints are NOT affected (rate limiter gates only these three unauth routes)
And: when Valkey is unavailable, Postgres fallback (evt_auth_rate_limit_window) enforces the limit
```

## AC-3: Single-use tokens for reset + magic-link

```gherkin
Given a valid, unconsumed password-reset token T was issued 2 minutes ago
When the user successfully completes password reset using T
Then fct_password_reset_tokens.consumed_at is set atomically in the same transaction
And a second POST /v1/auth/password-reset/confirm with T returns 400 {code:"TOKEN_CONSUMED"}
And an audit event iam.token.replay_blocked is emitted (not the success event again)

And: same semantics apply to magic-link tokens (fct_magic_link_tokens)
And: expired-but-unconsumed tokens return {code:"TOKEN_EXPIRED"} (distinct from replay)
And: reset token TTL is 15 min, magic-link TTL is 10 min (both tightened from prior defaults)
```

  <action>
    Register new request-kind node `iam.auth.rate_limit` in manifest. kind=request, tx=none (pre-auth hot path; ingest-node precedent), emits_audit=false at node-level (the rate-limit decision itself is hot); the route layer emits audit only when the limiter fires (denies the request) via run_node("audit.events.emit", ...).

    Handler logic (`node.rate_limit`):
    - Inputs: `{ip: str, endpoint: str, window_seconds: int, max_requests: int}`
    - Try Valkey INCR on key `authrl:{endpoint}:{ip}:{floor(now/window)}` with EXPIRE=window; if new count > max, return {allowed: False, retry_after: window_seconds - (now % window)}
    - Valkey unavailable: fall back to Postgres table `evt_auth_rate_limit_window` (ip TEXT, endpoint TEXT, window_start TIMESTAMP, count INT) with UPSERT + increment in a fresh conn (not caller's tx — counters are not transactional with the auth request)
    - Return {allowed: True, remaining: max - count} on pass

    New migration NNN_auth_rate_limit_window.sql creates evt_auth_rate_limit_window with PK (endpoint, ip, window_start); index on window_start for periodic GC.

    FastAPI dependency `auth_rate_limit(endpoint: str, max: int = 10, window: int = 60)`:
    - Pulls client IP from X-Forwarded-For (trusted proxy) or request.client.host
    - Calls run_node("iam.auth.rate_limit", ...)
    - If !allowed: emit audit iam.auth.rate_limited (category=security, outcome=blocked, scope={ip, endpoint}, details={window, max}) + raise HTTPException(429, detail={"code":"RATE_LIMITED","retry_after":retry_after})
    - Set Retry-After response header

    Apply as Depends to:
    - POST /v1/auth/login → (10, 60)
    - POST /v1/auth/magic-link/request → (5, 60)
    - POST /v1/auth/password-reset/request → (3, 60)

    Do NOT apply to authenticated endpoints — this limiter gates pre-auth only. Authenticated rate limiting is a separate concern (future).

    Avoid: blocking the auth request on a Valkey timeout — use short timeout + fail-open-to-Postgres-fallback, not fail-open-to-unlimited.
    Avoid: keying on user-supplied email (trivially bypassed by varying the email field).

    ```sql
    UPDATE fct_{x}_tokens
       SET consumed_at = CURRENT_TIMESTAMP
     WHERE id = $1 AND consumed_at IS NULL AND expires_at > CURRENT_TIMESTAMP
    ```
    Returns true only if exactly one row updated (atomic check-and-consume). Never raises; the boolean tells the caller which error to surface.

    Service: in `verify_and_consume_reset_token(conn, token_value)` / `verify_and_consume_magic_link(conn, token_value)`:
    - Look up token by hash
    - If not found → raise AuthError("token_invalid")
    - If expires_at < now → emit audit iam.token.expired; raise AuthError("token_expired")
    - If consumed_at is not null → emit audit iam.token.replay_blocked (scope=user_id, details={token_type, original_consumed_at}); raise AuthError("token_consumed")
    - Else call consume_token; if it returns False (race), emit replay_blocked and raise
    - Else proceed with reset/login flow

    - password_reset: expires_at = now + 15 min (prior default was likely 1 hour or longer)
    - magic_link: expires_at = now + 10 min

    Create ADR-031 documenting: single-use invariant, TTL choices (balance between UX and replay window), why we emit replay_blocked separately from expired (different signal to security monitoring).

    Avoid: deleting the token row on consume — we need the consumed_at audit trail for replay detection. Periodic GC (deleted_at based) cleans old rows later.
    Avoid: using SELECT-then-UPDATE without the atomic UPDATE-WHERE-consumed_at-IS-NULL pattern — race opens replay window.

- No UI changes (backend + migrations + tests only)
- No new SDK capabilities — the auth SDK already speaks to existing endpoints; 429 handling is added to transport if trivial, else left for the SDK owner
- No session absolute-max-TTL change (separate concern)
- No CSRF — cookies are SameSite=Lax + Secure + HttpOnly; same-origin API so CSRF risk is minimal for current shape
- No CAPTCHA / proof-of-work — rate limit alone is the hardening for this plan
- No adaptive / anomaly-based rate limiting — fixed window is enough for OSS hardening
