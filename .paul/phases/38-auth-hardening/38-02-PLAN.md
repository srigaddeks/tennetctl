<objective>
## Goal

Close the two rotation boundaries deferred from Plan 38-01: rotate the caller's
session_id on (a) successful password change and (b) successful TOTP enrollment.
On both paths, mint a fresh session, revoke the old one, and re-set the cookie
so the browser picks up the new session_id transparently.

## Purpose

Plan 38-01 closed session fixation on login. Password-change and MFA-enrollment
are the other two classic privilege-escalating boundaries (OWASP ASVS V3.2.1).
After a password change, the old session_id predates the new authentication
state — replay risk. After MFA enrollment, the old session was pre-MFA — its
effective auth strength just jumped. Rotate both.

## Output

- `rotate_on_privilege_escalation()` helper in 09_sessions/service.py
  (generalization of 38-01's `rotate_on_login`; takes a reason literal)
- `change_password` service returns new `(token, session_meta)` alongside the
  existing `other_revoked` count; route re-sets the cookie
- TOTP enroll-confirm path in 12_otp returns new `(token, session_meta)`;
  route re-sets the cookie
- Two new audit events: `iam.session.rotated{reason=password_change}` and
  `iam.session.rotated{reason=mfa_enrolled}` (existing event key, new reasons)
</objective>

<acceptance_criteria>

## AC-1: Session rotation on password change

```gherkin
Given an authenticated user with session_id S1
When they PATCH /v1/credentials/me with valid {current_password, new_password}
Then the response includes a new session token whose session_id S2 differs from S1
And the response Set-Cookie header carries the new token
And fct_sessions shows S1 revoked (revoked_reason = "rotated_on_password_change") and S2 active
And all OTHER sessions for the user are also revoked (existing 38-01 behavior preserved)
And an audit event iam.session.rotated{reason=password_change} is emitted
And the new session inherits user_id + org_id + workspace_id from the old one (no re-login)
```

## AC-2: Session rotation on TOTP enrollment confirmation

```gherkin
Given an authenticated user with session_id S1 who has just scanned a TOTP secret
When they POST /v1/auth/otp/totp/confirm with a valid 6-digit code
Then the enrollment succeeds (totp credential persisted)
And the response includes a new session token whose session_id S2 differs from S1
And the response Set-Cookie header carries the new token
And S1 is revoked (revoked_reason = "rotated_on_mfa_enrolled")
And an audit event iam.session.rotated{reason=mfa_enrolled} is emitted
And the existing iam.otp.totp.enrolled audit still fires (not replaced)
```

<boundaries>

## DO NOT CHANGE

- 38-01's `rotate_on_login` behavior on the signin path — it was just shipped and works; only generalize the helper, don't rewrite signin
- Existing `iam.otp.totp.enrolled` audit emit — preserve it; rotation audit is ADDITIONAL
- The "revoke all OTHER sessions" behavior in `change_password` — keep it; rotation is ADDITIONAL to (not instead of) sibling revocation
- TOTP enrollment validation logic itself (secret generation, code verification) — out of scope
- Rate-limit dependency on password-reset/request + magic-link/request routes — that's 38-01 work, stable

## SCOPE LIMITS

- No rotation on mid-session step-up (TOTP challenge during sensitive operations) — separate concern, separate future plan
- No rotation on passkey enroll / email-verify — same boundary class but lower priority (no password creds to escalate) and they have their own flows; follow-up plan if needed
- No UI changes — frontend auth hook should naturally pick up the new cookie; if the session context needs an invalidate hint, flag as DONE_WITH_CONCERNS
- No SDK changes — the Python/TS SDKs talk to these endpoints; the response body shape additions are backward-compatible (new optional fields)
- No new migrations — reuses existing fct_sessions + evt_audit infrastructure

</boundaries>
