<objective>
## Goal
Docker Compose stack (Postgres 16, MinIO, NATS JetStream, Qdrant, Valkey) running as `tennetctl_v2`, plus a custom SQL migrator that auto-tracks and applies migrations — the foundation every subsequent plan builds on.

## Purpose
The migrator is the single most important piece of infrastructure. Every phase after this just drops `.sql` files and they run. Docker Compose gives us the full infra stack from day one — no "I'll add that later" drift.

## Output
- `docker-compose.yml` with all 5 services running under `tennetctl_v2` project
- `scripts/migrator/` with async Python migration runner
- First migration: `00_schema_migrations` tracking table
- `requirements.txt` with minimal deps (asyncpg only for now)
- Passing integration tests against real Postgres
</objective>

## Key Conventions
- asyncpg for all DB access — no ORM, no SQLAlchemy
- Migration naming: YYYYMMDD_NNN_description.sql
- Schema naming: "NN_name" (e.g., "00_schema_migrations", "03_iam", "04_audit")

<task type="auto">

  <files>

  <action>
    **requirements.txt:**
    - asyncpg>=0.30.0 (only dependency needed for migrator)

    **scripts/migrator/runner.py:**

    Core functions:
    - `async def run_migrations(dsn: str, migrations_dir: str | None = None)` — main entry
    - `async def get_applied(conn) -> list[dict]` — return applied migrations with checksums
    - `def compute_checksum(content: str) -> str` — SHA256 hex digest
    - `def discover_migrations(migrations_dir: str) -> list[Path]` — sorted *.sql files

    Bootstrap logic:
    - First, ensure "00_schema_migrations" schema exists
    - Then ensure applied_migrations table exists:
      ```sql
      CREATE SCHEMA IF NOT EXISTS "00_schema_migrations";
      CREATE TABLE IF NOT EXISTS "00_schema_migrations".applied_migrations (
        id SERIAL PRIMARY KEY,
        filename TEXT NOT NULL UNIQUE,
        applied_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
        checksum TEXT NOT NULL
      );
      ```

    Migration loop:
    - Discover all .sql files in migrations_dir, sort by filename
    - Fetch already-applied set from DB
    - For each pending migration:
      - Read file content
      - Compute SHA256 checksum
      - If filename exists in applied BUT checksum differs → ERROR with clear message, exit
      - If filename exists in applied AND checksum matches → skip (log "already applied")
      - If filename NOT in applied → execute in transaction, then INSERT into applied_migrations
    - Log summary: "Applied N of M migrations"

    Default migrations_dir: `scripts/migrator/migrations/` (relative to project root)

    CLI entry (`if __name__ == "__main__"`):
    - `--dsn` flag (default from DATABASE_URL env var)
    - `--dir` flag (default migrations dir)
    - `--dry-run` flag (show what would be applied without applying)
    - Uses asyncio.run()

    **First migration file** (`00000000_000_create-migration-tracking.sql`):
    - This is a no-op/self-referential — the bootstrap logic creates the table,
      but this file exists so the migrator records itself as "applied" for consistency.
    - Content: just a comment explaining this is tracked by the bootstrap
    ```sql
    -- Migration tracking table bootstrap
    -- This table is created by the migrator bootstrap logic.
    -- This file exists so the migration is recorded in applied_migrations for consistency.
    SELECT 1;
    ```

    Avoid: Running migrations outside transactions, using ORM, ignoring checksum mismatches, swallowing errors
  </action>
  <verify>
    DATABASE_URL=postgresql://tennetctl:tennetctl_dev@localhost:5432/tennetctl python -m scripts.migrator.runner
    Then verify: psql -h localhost -U tennetctl -d tennetctl -c "SELECT * FROM \"00_schema_migrations\".applied_migrations;"
  </verify>
  <done>AC-2, AC-3, AC-4 satisfied: Tracking table created, migrations applied in order, checksums validated</done>
