# Sandbox Signal Builder API

**Base path:** `/api/v1/sb`
**Auth:** Bearer JWT required on all endpoints
**Permissions:** `sandbox.view`, `sandbox.create`, `sandbox.execute`
**Multi-tenant:** All endpoints require `org_id` as a query parameter. `tenant_key` is derived from the authenticated user's active organization.

---

## Overview

Signals are the atomic evaluation units of the sandbox pipeline. Each signal is a Python function stored in the database that accepts a dataset and returns a structured result. Signals are authored manually, generated by AI, and promoted through a lifecycle before being used in threat type expression trees.

**Lifecycle:** `draft` → `testing` → `validated` → `promoted` → `archived`

- **draft** — Initial authoring; not yet run against any test data
- **testing** — Golden test expectations have been defined; awaiting validation run
- **validated** — All golden test expectations pass; ready for promotion
- **promoted** — Actively used in threat type evaluations in production
- **archived** — Retired; preserved for history but excluded from new threat types

Status transitions are strict and forward-only except for `promoted` → `archived`. Editing the `python_source` property resets the status to `draft` and creates a new version.

**Execution model:** Signal Python source runs inside RestrictedPython + subprocess isolation. Signals cannot import arbitrary modules, make network calls, or access the database. The `evaluate` function contract is enforced at validation time.

---

## Signal Function Contract

All signals must expose an `evaluate` function with this exact signature. The execution engine calls `evaluate(dataset)` and validates the return structure before recording the result.

```python
def evaluate(dataset: dict) -> dict:
    """
    Args:
        dataset: The full parsed JSON payload from a Dataset record.

    Returns:
        dict with keys:
            result   (str)  — "pass", "fail", or "warning"
            summary  (str)  — Single-sentence human-readable summary
            details  (list) — List of per-check result objects
            metadata (dict) — Optional extra data for downstream consumers
    """
    return {
        "result": "pass",        # required: "pass" | "fail" | "warning"
        "summary": "...",        # required: non-empty string
        "details": [             # required: list (may be empty)
            {
                "check": "binding_name",
                "status": "pass",
                "message": "No wildcard resources found"
            }
        ],
        "metadata": {}           # optional: arbitrary dict
    }
```

**Validation rules enforced by the execution engine:**

- `result` must be exactly `"pass"`, `"fail"`, or `"warning"`
- `summary` must be a non-empty string
- `details` must be a list; each element must have `check` (string) and `status` (string)
- Functions that raise an unhandled exception are recorded as `result: "error"` and do not advance the signal's status
- Execution timeout: 30 seconds per signal run

## Architecture

- **Fact table:** `15_sandbox.22_fct_signals` — lean row with FK columns, status, version, workspace
- **Properties:** `15_sandbox.45_dtl_signal_properties` — EAV key-value pairs; `python_source` is stored here, not in the fact table
- **Test expectations:** `15_sandbox.49_dtl_signal_test_expectations` — golden test dataset/result pairs
- **Connector compatibility:** `15_sandbox.50_lnk_signal_connector_types` — which connector types a signal is designed for
- **Execution results:** ClickHouse (not PostgreSQL) — all run results are written to ClickHouse for analytics; only summaries are surfaced via the run history API

**Security:** `python_source` is **never returned** in list endpoints (`GET /signals`). It is only returned in the single-record detail endpoint (`GET /signals/{id}`) to prevent accidental exposure of large code payloads in list UIs and to ensure callers are making an explicit fetch.

**EAV property keys:**

| Key | Type | Description |
|-----|------|-------------|
| `name` | string | Display name |
| `description` | string | What the signal checks and why |
| `python_source` | string | Full Python source of the `evaluate` function |
| `source_prompt` | string | AI generation prompt (set by `/generate`; null for manual signals) |
| `tags` | string | Comma-separated tags |
| `usage_notes` | string | Notes for signal consumers about expected dataset structure |
| `parameter_schema` | string | JSON Schema string for parameterized signals (future) |
| `caep_event_type` | string | Corresponding CAEP event type code (e.g., `session-revoked`) |
| `risc_event_type` | string | Corresponding RISC event type code (e.g., `credential-compromise`) |

| `500` | Unexpected server error |

### PATCH /api/v1/sb/signals/{signal_id}

Update a signal. Creates a **new version** (version_number auto-increments). If `python_source` is included in `properties`, the status resets to `draft` on the new version. Property-only updates (no `python_source` change) preserve the current status.

**Permission:** `sandbox.create`

**Path params**

| Param | Type | Description |
|-------|------|-------------|
| `signal_id` | UUID | Signal ID |

**Query params**

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `org_id` | UUID | Yes | Organization scope |

**Request body** (all fields optional)

| Field | Type | Description |
|-------|------|-------------|
| `properties` | object | Partial property update — provided keys are upserted, omitted keys are unchanged |
| `connector_type_codes` | array of string | Replace connector type associations (full replacement, not additive) |
| `test_expectations` | array | Replace all test expectations (full replacement) |

```json
{
  "properties": {
    "python_source": "def evaluate(dataset: dict) -> dict:\n    # updated: also checks role bindings in namespaces\n    ...",
    "description": "Updated: now checks both cluster-level and namespace-level role bindings",
    "tags": "kubernetes,rbac,security,namespace"
  },
  "connector_type_codes": ["kubernetes"],
  "test_expectations": [
    {
      "dataset_id": "a9b0c1d2-e3f4-5678-abcd-901234567890",
      "expected_result": "fail",
      "expected_summary_contains": "wildcard"
    },
    {
      "dataset_id": "c1d2e3f4-a5b6-7890-cdef-123456789012",
      "expected_result": "pass"
    }
  ]
}
```

**Response** `200 OK` — Updated signal object with incremented `version_number`

**Business rules:**
- Updating `python_source` resets `status_code` to `draft` on the new version
- Updating only descriptive properties (name, description, tags, etc.) does not reset the status
- `test_expectations` in the request body fully replaces the existing expectations — omitting it leaves them unchanged
- A signal in `promoted` status can be patched; the promoted version is preserved in history and the new version starts at `draft`

**Error codes**

| Status | Condition |
|--------|-----------|
| `400` | `python_source` does not define an `evaluate` function |
| `401` | Missing or expired JWT |
| `403` | JWT lacks `sandbox.create` or signal belongs to different org |
| `404` | Signal ID or a `dataset_id` in `test_expectations` not found |
| `422` | Malformed UUID or invalid field values |
| `500` | Unexpected server error |

### DELETE /api/v1/sb/signals/{signal_id}

Soft-delete a signal. Sets `is_active = false`. Fails if the signal is in `promoted` status and referenced by an active threat type expression tree.

**Permission:** `sandbox.create`

**Path params**

| Param | Type | Description |
|-------|------|-------------|
| `signal_id` | UUID | Signal ID |

**Query params**

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `org_id` | UUID | Yes | Organization scope |

**Response** `204 No Content`

**Business rules:**
- `promoted` signals referenced by active threat types cannot be deleted; archive the signal and update the threat type first
- Run history in ClickHouse is preserved after deletion

**Error codes**

| Status | Condition |
|--------|-----------|
| `401` | Missing or expired JWT |
| `403` | JWT lacks `sandbox.create` or signal belongs to different org |
| `404` | Signal ID not found |
| `409` | Signal is `promoted` and referenced by one or more active threat types |
| `422` | Malformed UUID |
| `500` | Unexpected server error |

## AI Signal Generation

### POST /api/v1/sb/signals/generate

Generate a signal from a natural language prompt using a LangGraph AI agent. The agent inspects the sample dataset schema, drafts Python source implementing the `evaluate` contract, and creates a new signal record in `draft` status. The generated `source_prompt` is stored as an EAV property for traceability.

**Permission:** `sandbox.create`

**Query params**

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `org_id` | UUID | Yes | Organization scope |

**Request body**

| Field | Type | Required | Validation | Description |
|-------|------|----------|------------|-------------|
| `prompt` | string | Yes | 10–2000 chars | Natural language description of the check to implement |
| `connector_type_code` | string | Yes | Must exist in `03_dim_connector_types` | Constrains the AI to the correct dataset schema |
| `workspace_id` | UUID | Yes | Must exist within org | Workspace for the generated signal |
| `sample_dataset_id` | UUID | No | Must exist within org | Dataset provided to the agent as schema reference; improves generation quality |

```json
{
  "prompt": "Check that no Kubernetes pods are running as the root user. A pod is running as root if securityContext.runAsUser is 0, or if runAsNonRoot is not explicitly set to true.",
  "connector_type_code": "kubernetes",
  "workspace_id": "e5f6a7b8-c9d0-1234-efab-345678901234",
  "sample_dataset_id": "a9b0c1d2-e3f4-5678-abcd-901234567890"
}
```

**Response** `201 Created` — Full signal object

```json

  "signal_code": "ai-gen-k8s-no-root-pods",

  "status_code": "draft",
  "workspace_id": "e5f6a7b8-c9d0-1234-efab-345678901234",
```

**Business rules:**
- `signal_code` is auto-generated as `ai-gen-{slug}` where `{slug}` is derived from the prompt
- If `sample_dataset_id` is provided, the agent schema-inspects the payload to produce more accurate field access patterns
- The AI agent does not run or validate the generated code; use `POST /signals/{id}/validate` to run it against test data
- Generation typically completes within 5–15 seconds depending on prompt complexity

**Error codes**

| Status | Condition |
|--------|-----------|

| `403` | JWT lacks `sandbox.create` |
| `404` | `workspace_id` or `sample_dataset_id` not found |
| `422` | Invalid `connector_type_code` |
| `429` | AI generation rate limit exceeded (max 20 per hour per org) |
| `500` | AI agent error or generation failed |
| `503` | AI generation service unavailable |

## Signal Validation

### POST /api/v1/sb/signals/{signal_id}/validate

Execute the signal against all golden test expectations. Each expectation runs the `evaluate` function with its linked dataset and compares the actual result to the expected result. If all expectations pass, the signal's status advances to `validated`. If any fail, the status remains unchanged and failures are returned.

**Permission:** `sandbox.execute`

**Path params**

| Param | Type | Description |
|-------|------|-------------|
| `signal_id` | UUID | Signal ID |

**Query params**

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `org_id` | UUID | Yes | Organization scope |

**Request body:** None required.

**Response** `200 OK` — All tests pass


**Status advancement logic:**

| Condition | Resulting status |
|-----------|-----------------|
| All tests pass | `validated` |
| Any test fails | `testing` (unchanged if already in testing) |
| Signal has no test expectations | Returns error — cannot validate without expectations |
| Signal's `evaluate` raises an exception | Result recorded as `error`; status stays `draft` |

**Business rules:**
- Execution runs inside RestrictedPython + subprocess isolation; the signal cannot access the database or network
- Validate advances to `validated` but does not promote — use PATCH to set status to `promoted` (or a dedicated promote endpoint if implemented)
- All run results are written to ClickHouse and visible in `GET /signals/{id}/runs`

**Error codes**

| Status | Condition |
|--------|-----------|
| `401` | Missing or expired JWT |
| `403` | JWT lacks `sandbox.execute` or signal belongs to different org |
| `404` | Signal ID not found |
| `409` | Signal has no test expectations defined — add expectations first |
| `422` | Malformed UUID |
| `500` | Execution sandbox error |

## Run History

### GET /api/v1/sb/signals/{signal_id}/runs

List execution history for a signal. Run records are written to ClickHouse by the execution engine and surfaced here. Includes both validation runs and production evaluation runs.

**Permission:** `sandbox.view`

**Path params**

| Param | Type | Description |
|-------|------|-------------|
| `signal_id` | UUID | Signal ID |

**Query params**

| Param | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `org_id` | UUID | Yes | — | Organization scope |
| `result` | string | No | — | Filter by result: `pass`, `fail`, `warning`, `error` |
| `sort_dir` | string | No | `desc` | `asc` or `desc` (ordered by `executed_at`) |
| `limit` | integer | No | `50` | 1–500 |
| `offset` | integer | No | `0` | >= 0 |
